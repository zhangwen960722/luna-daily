# Workflow: Phase 3 - 规格生成 (Make Specs)

此工作流是**最后一步**。前两步的思考和设计已经完成，现在是"组装"和"输出"。

## 步骤 1: 关键事实提取 (Fact Extraction)

> **Goal**: 在写 PRD 之前，先从 RDD 中提取一份"不能翻错的关键事实清单"。PRD 写完后逐条核对，零遗漏。

**输入**: Phase 1 的 RDD

**输出格式**（以超级运价为例，新模块同理）:
1. 核心计算规则（如：公布价 = 成本底价 + 利润策略，系统自动算，人工可覆盖）
2. 数据生命周期（如：运价行挂周批次，与船期同频，查询只取 CURRENT 批次）
3. 实体间联动（如：包税方式 → 关务段税金项启用/禁用）
4. 匹配与过滤逻辑（如：无匹配运价行 → 标"暂未报价"并展示渠道，不屏蔽）
5. 边界条件（如：附加费分两节点触发，运价查询阶段只匹配产品类型）
6. 继承与覆盖规则（如：计费规则客户覆盖 NULL=继承默认）

> **原则**：这份清单是 PRD 的"防翻车指南"。不要求穷尽，但必须覆盖所有跨实体的联动逻辑和容易理解偏差的设计决策。

---

## 步骤 2: 内容映射 (Content Mapping)

> **Goal**: 明确 RDD 和 Architecture 中哪些内容进 PRD 的哪些章节，避免遗漏或重复。

**输入**:
- Phase 1 的 RDD
- Phase 2 的 Architecture（数据模型 + 流程图 + IA 设计 + 外部接口清单）

**映射关系**:

| 来源 | 内容 | → PRD 章节 |
|------|------|-----------|
| RDD Section 1 | 核心洞察、痛点、JTBD、业务价值表 | → 1. 需求定义 |
| RDD Section 6 | MVP 范围 + 明确不做清单 | → 1.3 范围与边界 |
| RDD Section 2 | 角色（从各 User Story 提取）、典型场景（Epic 核心场景） | → 2. 用户与场景 |
| RDD Section 5 | 功能清单 + 优先级 | → 3. 功能清单与本期范围 |
| RDD Section 2 Epic 核心场景 + 业务规则 | 按 Epic 整理为流程步骤 + 规则编号表（R01/R02...） | → 4. 流程与规则 |
| Architecture 业务流程图 | 转换为 To-Be 流程步骤（编号步骤+责任人+产出） | → 4.1 业务流程 |
| RDD Section 4 | 计算示例（作为附录参考） | → 4. 流程与规则（附录） |
| Architecture IA 设计 | 菜单结构、页面清单 | → 5. 信息架构 |
| RDD 实体字段表 + Architecture Schema | 页面级的字段字典（非 DB 字段，是页面表单/列表字段） | → 6. 页面与字段 |
| RDD + Architecture | 权限角色、配置项 | → 7. 权限与配置 |
| Architecture 外部接口清单 | 接口定义（本模块依赖的外部系统接口） | → 8. 接口与数据 |
| RDD Section 3 | 验收标准 | → 9. 验收与上线 |

> **原则**：RDD 提供"做什么、为什么"，Architecture 提供"怎么做、什么结构"，PRD = 两者合并 + 给开发看的落地细节。

## 步骤 3: 生成PRD (PRD Generation)

1. 按映射关系，将 RDD 和 Architecture 内容填入 `templates/prd-template.md`。
2. 确保所有业务规则都有编号 (R01, R02...)。
3. 章节标注 `【条件必填】` 的，如不适用写"无/不涉及"，**不可直接删除章节**。
4. 检查 PRD 是否自包含——开发不需要翻阅 RDD 就能理解全部内容。

**质量检查**:

- [ ] 是否有文档基础信息（版本/状态/评审人）？
- [ ] 业务规则是否编号（R01, R02...）？
- [ ] 验收标准是否可测试？
- [ ] 条件必填章节是否已填写或标注"无"？
- [ ] 是否包含权限说明？
- [ ] 是否包含异常处理？

**RDD 一致性校验**（用 Step 1 的关键事实清单逐条核对）:
- [ ] 核心计算规则是否正确传递？
- [ ] 数据生命周期是否正确传递？
- [ ] 实体间联动是否正确传递？
- [ ] 匹配与过滤逻辑是否正确传递？
- [ ] 边界条件是否正确传递？
- [ ] 继承与覆盖规则是否正确传递？

## 步骤 4: 生成前端Demo (Optional)

> **Goal**: 将PRD转化为可视化、可交互的原型代码。

### 4.1 技术栈

基于项目实际技术栈（当前为 Vue 3 + Element Plus CDN 模式），不使用 React/Tailwind。

### 4.2 交互功能要求（硬性规则）

**Demo 不是静态图，所有按钮必须有真实响应：**

- **新增**：打开弹窗 → 填写表单 → 保存后 `push` 到数据数组，表格即时刷新
- **编辑**：点击行级编辑按钮 → 回填当前行数据到表单 → 保存后 `splice` 替换原数组项
- **删除/禁用/启用**：`splice` 删除或切换行内状态字段
- **筛选/搜索**：computed 属性过滤数据数组
- **保存校验**：必填项为空时 `ElMessage.warning` 提示，不关闭弹窗

**禁止事项：**
- 禁止保存按钮只 `ElMessage.success` 然后关弹窗，不写数组
- 禁止编辑按钮无 `@click` 处理
- 禁止按钮点击后无任何反馈

### 4.3 数据模型映射（硬性规则 — 禁止凭记忆生成）

**每条表单字段必须先找到 RDD 实体字段表 + Architecture Schema 中的定义，再做 UI。**

生成每个页面前，强制执行以下检查：

1. **实体数量对齐**：读 RDD Section 2 该实体的 `### 实体名` 标题。一个实体 = 一个弹窗/Tab/表格。RDD 说有三个实体（如三张时效表），就必须做三个 Tab，不能合并成一个。
2. **字段名对齐**：表单 label 和表格列头必须与 RDD 字段表的"字段"列一致，不能自己编造（如 RDD 写"开船至送仓"就不能写成"预估送仓时效"）。
3. **字段类型映射**：对照 Architecture Schema 的"类型"列，映射到正确的表单控件：

| Schema 类型 | 表单控件 |
|---|---|
| `String` / `Varchar` / `Text` | `el-input` |
| `Date` / `DateTime` / `Timestamp` | `el-date-picker` |
| `Int` / `Integer` / `BigInt` | `el-input-number`（:min/:max 约束） |
| `Decimal(n,2)` | `el-input-number`（:precision="2"） |
| `Boolean` / `TinyInt` | `el-switch` |
| `Enum` | `el-select` + options 来自 RDD 枚举值 |
| `JSON` | 动态列表（`v-for` 数组 + 添加/删除行按钮） |

4. **关系映射**：RDD 字段表的"关系"列决定表单结构：
   - `1:1` → 表单字段直接嵌在编辑弹窗内
   - `1:N` → 弹窗内用表格 + 新增/编辑子弹窗

**禁止事项：**
- 禁止不看 Schema 类型就直接用 `el-input`（文本框万能是错的）
- 禁止把 JSON 字段（如 `{"广州仓":"2月1日"}`）硬编码成固定列（如 `截单(广州) | 截单(汕头)`）
- 禁止漏掉 RDD 中定义的子实体

### 4.4 子页面跳转

列表页中的子页面入口（如"时效配置""重量段""成本项""计费规则"按钮），当前阶段做占位处理：`ElMessage.info('进入 XX 详情页')`。

### 4.5 代码规范

1. 使用 `vue.global.prod.js` CDN（非 dev 版）。
2. 使用传统 `(function(){ var app = Vue.createApp({...}) })()` 语法，不用 `const{createApp}=Vue` 解构。
3. 枚举/标签映射提取到方法外部的查找表对象（如 `var typeLabel = { field: '字段匹配' }`），避免在模板中用三元表达式。
4. 编辑回填需要保留原始 `_raw` 对象，存表单状态（枚举 key）而非表格显示文本（枚举 value）。
5. **禁止自闭合组件标签**：CDN 模式下 Vue 自定义组件（`el-table-column`、`el-option`、`el-tag`、`el-button` 等 Element Plus 组件）不支持自闭合写法 `<el-tag />`。必须写完整闭合标签 `<el-tag ...></el-tag>`。只有原生 HTML 标签（`<br>`、`<img>`、`<input>`）可以用自闭合。违者会导致表格列嵌套挤压。

## 步骤 5: 交付与归档 (Delivery)

1. 将最终PRD保存到`prds/[模块名]/PRD`
- **文件示例**:
  - `prds/超级运价/PRD.md` (产品需求文档)
2. 将Demo代码保存到 `demo/[模块名称]/demo`.
- **文件示例**:
  - `demo/超级运价/demo.md` (前端demo代码)
3. 输出一份《验收标准清单(AC List)》供测试使用。

**Done**: 任务完成！
