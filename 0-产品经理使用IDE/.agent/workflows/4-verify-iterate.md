# Workflow: Phase 4 - 验证与迭代 (Verify & Iterate)

此工作流用于**质量保证阶段**。PRD输出不是终点，验证通过才能交付。

**输入**: 全部交付物 — 需求背景 + 用户需求(RDD) + 数据设计 + PRD + 原型 + AC List

> **模块聚焦**：验证默认限定当前模块的 7 份文档链路。若用户要求跨模块验证，按 CLAUDE.md §输入文件范围定义 → 工作模式判断 扩展范围。

## 步骤 1: 文档链路一致性检查 (Document Chain Verification)

> **Goal**: 按文档链路逐组对照，确保 7 份文档之间逻辑闭环、数据一致、无遗漏无矛盾。

---

### 1. 需求背景 ↔ 用户需求(RDD)

- [ ] 用户需求(RDD)的核心洞察是否回应了需求背景中记录的原始痛点？方向是否跑偏？
- [ ] 需求背景中提到的约束条件（时间/资源/技术栈）是否在用户需求(RDD)的 NFR 或 MVP 方案中体现？

### 2. 用户需求(RDD) ↔ 数据设计

- [ ] 用户需求(RDD)的实体字段表是否在数据设计中都有对应（表/子表/JSON 字段/纯逻辑实体）？
- [ ] 用户需求(RDD)的业务规则（R01/R02...）是否在数据设计中通过约束/索引/状态机得到支撑？
- [ ] 实体数量是否对齐？（用户需求(RDD)有 N 个实体 → 数据设计有 N 个表映射）

### 3. 数据设计 ↔ PRD

- [ ] PRD 字段定义与数据设计对齐？（字段名、类型、必填、枚举值一致，无遗漏或矛盾）
- [ ] PRD 状态机与数据设计状态流转定义一致？
- [ ] PRD 外部接口清单与数据设计的外部依赖匹配？

### 4. 用户需求(RDD) ↔ PRD

- [ ] 用户需求(RDD)的所有验收标准（AC）是否都在 PRD 中有对应的验收项？
- [ ] 用户需求(RDD)的业务规则（R01/R02...）是否完整传递到 PRD 的 Section 6？编号是否一致？
- [ ] 用户需求(RDD)的功能清单是否完整映射到 PRD 的功能清单与页面映射？

### 5. PRD ↔ 原型（如有）

**文档对齐**（PRD 规格 vs 原型实现）：
- [ ] 页面覆盖 — PRD 每个功能点是否有对应页面？
- [ ] 数据模型对齐 — 字段标签/类型/实体数量是否与 PRD 一致？
- [ ] 按钮行为 — 新增/编辑/删除/冻结是否真实操作数据？
- [ ] 必填校验/筛选搜索 — 是否生效？
- [ ] 异常状态 — 空状态/无匹配/错误提示是否覆盖？
- [ ] 冻结只读 + 状态二次确认 — 是否符合约定？

**前端规范**（对照 frontend-designer §必须遵守的约束 + §弹窗模板 + §表格列宽规范 逐项检查，不再逐条在此重复）

### 6. PRD ↔ AC List

- [ ] AC List 是否从 PRD 验收标准中提取？是否逐条可追溯？
- [ ] 优先级标注是否正确？（P0 必测 / P1 重要 / P2 可选）
- [ ] 是否补充了边界情况和异常场景的测试项？

### 7. 全局检查

- [ ] 状态流转是否闭环？（正向+逆向+异常路径均覆盖）
- [ ] 敏感数据是否脱敏？（手机号、身份证）
- [ ] 权限矩阵是否覆盖所有角色？
- [ ] 所有文档的模块名、日期、版本号是否一致？

## 步骤 2: 问题回溯 (Backtracking)

> **Goal**: 如果发现问题，回到对应阶段修正。

根据问题类型，回到对应Workflow：

| 问题类型         | 回到阶段        | 示例                           |
| :--------------- | :-------------- | :----------------------------- |
| **需求理解错误** | Phase 1         | 发现做的不是用户真实痛点       |
| **数据模型缺陷** | Phase 2 - 步骤1 | ER关系设计错误，缺少关键字段   |
| **边界情况遗漏** | Phase 2 - 步骤2 | 并发场景未覆盖，状态机有死循环 |
| **IA设计问题**   | Phase 2 - 步骤4 | 菜单层级不合理、页面遗漏       |
| **外部接口遗漏** | Phase 2 - 步骤3 | 缺少外部模块的接口依赖清单     |
| **PRD格式问题**  | Phase 3 - 步骤3 | 缺少章节、格式不规范           |
| **原型实现问题** | Phase 3 - 步骤4 | 弹窗组件错误、交互不符合规范   |

**修正后，必须重新执行本阶段（Phase 4）验证**。

## 步骤 3: 最终交付 (Delivery)

> **Goal**: 确认无误后，正式归档。

1. PRD 留在模块目录 `drafts/[模块名]/`，日期前缀标记为定稿日期。
2. 输出AC List供测试使用：
   - 基于Phase 1的 用户需求(RDD) 中的"验收标准"章节
   - 补充Phase 2发现的边界情况
   - 标注优先级（P0必测、P1重要、P2可选）
3. 变更记录追加遵循 CLAUDE.md §日志与变更追踪规则

## 步骤 4: 用户确认 (User Confirmation)

询问用户：

1. PRD是否符合预期？是否需要调整？
2. 原型（如有）是否需要优化？
3. 是否可以正式交付给开发团队？

**Done**: 任务完成，进入开发排期！
