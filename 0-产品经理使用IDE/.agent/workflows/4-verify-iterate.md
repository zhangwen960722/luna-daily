# Workflow: Phase 4 - 验证与迭代 (Verify & Iterate)

此工作流用于**质量保证阶段**。PRD输出不是终点，验证通过才能交付。

**输入**: 全部交付物 — 需求背景 + 用户需求(RDD) + 数据设计 + PRD + 原型 + AC List

## 步骤 1: 文档链路一致性检查 (Document Chain Verification)

> **Goal**: 按文档链路逐组对照，确保 6 份文档之间逻辑闭环、数据一致、无遗漏无矛盾。

---

### 1. 需求背景 ↔ 用户需求(RDD)

- [ ] 用户需求(RDD)的核心洞察是否回应了需求背景中记录的原始痛点？方向是否跑偏？
- [ ] 需求背景中提到的约束条件（时间/资源/技术栈）是否在用户需求(RDD)的 NFR 或 MVP 方案中体现？

### 2. 用户需求(RDD) ↔ 数据设计

- [ ] 用户需求(RDD)的实体字段表是否在数据设计中都有对应（表/子表/JSON 字段/纯逻辑实体）？
- [ ] 用户需求(RDD)的业务规则（R01/R02...）是否在数据设计中通过约束/索引/状态机得到支撑？
- [ ] 实体数量是否对齐？（用户需求(RDD)有 N 个实体 → 数据设计有 N 个表映射）

### 3. 数据设计 ↔ PRD

- [ ] PRD 字段定义与数据设计对齐？（字段名、类型、必填、枚举值一致，无遗漏或矛盾）
- [ ] PRD 状态机与数据设计状态流转定义一致？
- [ ] PRD 外部接口清单与数据设计的外部依赖匹配？

### 4. 用户需求(RDD) ↔ PRD

- [ ] 用户需求(RDD)的所有验收标准（AC）是否都在 PRD 中有对应的验收项？
- [ ] 用户需求(RDD)的业务规则（R01/R02...）是否完整传递到 PRD 的 Section 6？编号是否一致？
- [ ] 用户需求(RDD)的功能清单是否完整映射到 PRD 的功能清单与页面映射？

### 5. PRD ↔ 原型（如有）

- [ ] **页面覆盖** — PRD 的每个功能点是否有对应页面？页面数量是否与功能清单匹配？
- [ ] **数据模型对齐** — 每个表单字段是否在 PRD 字段表中有对应？字段标签是否与 PRD 一致？
- [ ] **字段类型对齐** — Date 用 `el-date-picker`？Enum 用 `el-select`？Boolean 用 `el-switch`？JSON 用动态列表？
- [ ] **实体数量对齐** — PRD 定义的 1:1 和 1:N 子实体是否都有对应的弹窗/Tab/表格？
- [ ] 新增按钮 → 是否打开弹窗？保存后是否写入数据数组、表格即时刷新？
- [ ] 编辑按钮 → 是否有 `@click` 处理？是否回填当前行数据？保存后是否替换数组项？
- [ ] 删除/禁用/启用按钮 → 是否有 `@click` 处理？是否真实修改数据？
- [ ] 弹窗保存 → 必填项是否有校验（非空提示）？校验失败是否阻止关闭？
- [ ] 筛选/搜索 → 下拉选择后表格数据是否过滤？
- [ ] 禁止项 → 无只弹 toast 不写数组的保存、无无 `@click` 的空按钮、无文本框代替日期选择器
- [ ] **标签闭合** — 无 `<el-table-column .../>` 自闭合写法（CDN 模式白屏）
- [ ] Mock 数据是否符合真实业务场景？
- [ ] 异常状态 UI 是否覆盖？（空状态、无匹配结果、错误提示）
- [ ] **弹窗方式** — 是否用 `el-dialog` 居中弹窗（非 `el-drawer`）？
- [ ] **弹窗重置** — 是否有 `destroy-on-close`？
- [ ] **表单校验** — `el-form-item` 是否均带 `prop`？
- [ ] **冻结只读** — 已冻结数据编辑弹窗是否自动切换为只读（标题变"查看XX"、确定按钮隐藏）？
- [ ] **状态二次确认** — 禁用/启用/冻结是否有 `ElMessageBox.confirm` 二次确认？
- [ ] **搜索规范** — 搜索字段描述是否在 `placeholder` 内（非外部 label）？
- [ ] **表格列宽** — 常规列是否用 `min-width`？操作列是否 `fixed="right"`？

### 6. PRD ↔ AC List

- [ ] AC List 是否从 PRD 验收标准中提取？是否逐条可追溯？
- [ ] 优先级标注是否正确？（P0 必测 / P1 重要 / P2 可选）
- [ ] 是否补充了边界情况和异常场景的测试项？

### 7. 全局检查

- [ ] 状态流转是否闭环？（正向+逆向+异常路径均覆盖）
- [ ] 敏感数据是否脱敏？（手机号、身份证）
- [ ] 权限矩阵是否覆盖所有角色？
- [ ] 所有文档的模块名、日期、版本号是否一致？

## 步骤 2: 问题回溯 (Backtracking)

> **Goal**: 如果发现问题，回到对应阶段修正。

根据问题类型，回到对应Workflow：

| 问题类型         | 回到阶段        | 示例                           |
| :--------------- | :-------------- | :----------------------------- |
| **需求理解错误** | Phase 1         | 发现做的不是用户真实痛点       |
| **数据模型缺陷** | Phase 2 - 步骤1 | ER关系设计错误，缺少关键字段   |
| **边界情况遗漏** | Phase 2 - 步骤2 | 并发场景未覆盖，状态机有死循环 |
| **IA设计问题**   | Phase 2 - 步骤4 | 菜单层级不合理、页面遗漏       |
| **外部接口遗漏** | Phase 2 - 步骤3 | 缺少外部模块的接口依赖清单     |
| **PRD格式问题**  | Phase 3 - 步骤3 | 缺少章节、格式不规范           |
| **原型实现问题** | Phase 3 - 步骤4 | 弹窗组件错误、交互不符合规范   |

**修正后，必须重新执行本阶段（Phase 4）验证**。

## 步骤 3: 最终交付 (Delivery)

> **Goal**: 确认无误后，正式归档。

1. PRD 留在模块目录 `drafts/[模块名]/`，日期前缀标记为定稿日期。
2. 输出AC List供测试使用：
   - 基于Phase 1的 用户需求(RDD) 中的"验收标准"章节
   - 补充Phase 2发现的边界情况
   - 标注优先级（P0必测、P1重要、P2可选）
3. 在对应模块的 `变更记录.md` 追加：`| YYYY-MM-DD | 验证通过，交付开发 | — |`

## 步骤 4: 用户确认 (User Confirmation)

询问用户：

1. PRD是否符合预期？是否需要调整？
2. 原型（如有）是否需要优化？
3. 是否可以正式交付给开发团队？

**Done**: 任务完成，进入开发排期！
