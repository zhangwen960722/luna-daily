# Workflow: Phase 1 - 需求洞察 (Insight)

此工作流用于**需求分析阶段**。不要直接开始写方案，先搞清楚"为什么做"和"为谁做"。

## 步骤 0: 上下文加载 (Context) ★ 不可跳过

> **Goal**: 加载所有已有的分析和背景文档，避免遗漏已确认的信息。

在开始任何需求分析前，**必须先读取以下文件**：

| 优先级 | 文件 | 用途 |
|:---|:---|:---|   
| 必读 | `context/`目录下所有 `.md`| 项目目标、技术约束（团队规模、订单量级、实时性要求），这些约束决定方案可行性 |
| 必读 | `analysis/` 目录下所有文件 | 获取到一些精准的业务背景资料 |
| 必读 | `drafts/` 目录下所有 `.md` | 已有分析文档（业务分析、需求调研），包含已确认的实体模型、状态机、场景决策 |
| 参考 | `.agent/rules/project-rule.md` | 协作规则和表达偏好 |
**为什么这一步不可跳过**：
- 用户已经和 AI 进行过多轮深度对话，大量信息已沉淀在这些文档中
- 如果 AI 凭记忆直接生成用户需求(RDD)，必然遗漏已确认的功能场景（如货物查询、任务大厅等）
- 用户不想重复沟通同样的内容

## 步骤 0.5: MECE 穷尽 + 集合关系检查 ★

> **Goal**: 在进入需求分析前，先穷尽场景/状态/关系，标注已覆盖和遗漏。

**触发**：任何涉及状态枚举、场景分支、实体绑定的讨论。

**MECE 穷尽检查**：

| # | 检查 |
|---|------|
| 1 | 你说的 N 种情况，有没有第 N+1 种？（≥、=、< 永远是三种，不是两种） |
| 2 | 每种情况的业务处理方式是什么？ |
| 3 | A→B 之间有没有中间/过渡状态？ |

**集合关系穷尽检查**（涉及两个集合 A 与 B 的包含/绑定关系时）：

| 关系 | 如何处理 |
|------|---------|
| A 全含 B | 正常流程 |
| A 不含 B | 拒绝/提示 |
| A 部分含 B | ← 最容易遗漏。B 中属于 A 的部分正常处理，不属于 A 的部分：留在原位/移走/报警？ |

**输出格式**：穷尽矩阵，标注"本期已覆盖 / ❌ 遗漏待补"

## 步骤 0.6: 正向→逆向对称检查 ★

> **Goal**: 每一个正向流程，必须同步产出逆向清单。

**触发**：任何涉及"下单/生成/创建/提交/确认/锁定/绑定"等正向动作。

**检查项**：

| # | 问题 |
|---|------|
| 1 | 什么条件触发逆向？（取消/撤回/拦截/回退/解绑） |
| 2 | 逆向后关联实体如何处理？ |
| 3 | 逆向后能否重新正向？如果能，是新创建还是恢复旧状态？ |
| 4 | 什么因素变化会导致必须走逆向再重新正向？（如：地址变了、尺寸变了） |

**输出**：逆向场景清单 + 每个逆向场景的影响范围说明。

## 步骤 1: 需求澄清与追问 (Clarify)

> **Goal**: 识别并拦截伪需求，挖掘业务价值。

请调用 **Skill: `requirement-clarifier`** 执行以下操作：

**调用方式**: 阅读`.agent/skills/requirement-clarifier/SKILL.md`，按照其指引执行

**输入**: Step 0 已加载的全部上下文（`context/`、`analysis/`、`drafts/` 下所有文件）+ 用户在本轮对话中描述的需求（一句话需求/需求文档路径/功能描述等）

**执行步骤**:

1. 分析用户输入的原始需求。
2. 识别是否存在 "X-Y Problem"（即用户给的是解决方案而非问题）。
3. 使用 JTBD 和 USM 框架进行结构化追问（如果信息不全）。

**输出期望**:

- 如果需求模糊，提出3个关键问题给用户，直到信息足够
- 如果需求清晰，转至步骤2生成用户需求(RDD)

## 步骤 2: 需求定义 (Define)

> **Goal**: 将模糊需求转化为结构化的定义。

当信息收集完整后，使用 `requirement-clarifier` Skill 中定义的模板，输出《用户需求(RDD)》。

**必须包含**:

- 核心洞察 (Insight)
- 业务全景图（角色/链路/实体依赖）
- 按业务流程组织的功能设计（每条流程含实体字段、业务规则、核心场景、相关 AC）
- 验收标准总览（按流程分组）
- NFR
- 计算示例
- 功能清单 & 分期
- MVP 方案与建议

### 质量检查（生成用户需求(RDD)后必须自检）

- [ ] 是否识别出真实痛点（非伪需求）
- [ ] 业务价值是否量化（High/Medium/Low）
- [ ] 是否有业务全景图（角色×节奏、端到端链路、实体依赖）
- [ ] 每个数据实体是否有字段表（字段/类型/必填/说明）
- [ ] 是否按业务流程组织而非按实体+Epic 堆叠
- [ ] 每条流程是否标注触发条件、频率、前置依赖
- [ ] 验收标准是否可测试且可按流程追溯

### 关于成功标准

**0→1 项目不需要强行量化 KPI**。成功标准从两个维度定义：
- **员工视角**：核心操作路径是否缩短？是否从多系统切换变为单一系统闭环？
- **管理视角**：流程是否合规可控？数据是否可视化？决策是否有据可依？

## 步骤 3: 归档 (Archive)

将生成的用户需求(RDD)保存到 `drafts/[模块名]/YY-MM-DD-用户需求.md`，并询问用户是否进入下一阶段（方案架构）。
- **文件示例**:
  - `drafts/超级运价/2026-05-01-用户需求.md` (需求定义卡片)
