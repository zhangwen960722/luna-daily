# Workflow: Phase 2 - 方案架构 (Architecture)

此工作流用于**方案设计阶段**。在写文档前，先确保地基稳固。

> **模块聚焦**：本阶段承接 Phase 1 的模块上下文，文件加载默认限定当前模块。若涉及跨模块依赖（如引用外部模块的实体），按 CLAUDE.md §输入文件范围定义 → 工作模式判断 确定扩展范围。

## 步骤 1: 实体建模 (Data Modeling)

> **Goal**: 将用户需求(RDD)的业务实体转化为数据库 Schema。

**调用方式**: 阅读 `.agent/skills/data-modeler/SKILL.md`，按照其指引执行。

**输入**:

- Phase 1 的 用户需求(RDD)（需求定义卡片）— 所有带字段表的实体，包括业务全景图中的数据实体和各 Epic 内部引入的新实体
- 用户需求(RDD)中的实体关系描述（ASCII 关联链路图、关联说明）

**执行步骤**:

1. 读取 用户需求(RDD)，提取所有带字段表的实体（业务全景图中的数据实体 + Epic 内新引入实体），列出完整的实体清单。
2. 对每个实体补充数据库层设计：主键策略、审计字段（created_by/updated_at/version）、索引建议、外键关系。
3. 确定实体×表的映射关系：哪些实体是独立表、哪些是子表、哪些字段存 JSON、哪些是纯逻辑实体（如引用外部模块的港口）。
4. 按 data-modeler 规范输出字段清单（字段名/类型/必填/约束/枚举值/默认值）。
5. 绘制 ER 图（Mermaid 格式），标注关系类型（1:1 / 1:N / N:N）和外键。

**输出期望**:

- 实体清单 × 表映射（哪个实体对应哪张表，纯逻辑实体标注"无独立表"）
- 每张表的字段清单（含主键、审计字段、索引、外键）
- ER 关系图（Mermaid 格式）
- 关键设计说明（软删除策略、乐观锁、JSON 字段的使用场景）

## 步骤 2: 流程编排 (Process Design)

> **Goal**: 确定业务流转。

**输入**:

- 步骤1生成的数据模型
- 业务正向流程描述

**执行步骤**:

1. 绘制 Mermaid 流程图（包含正向流程 + 逆向流程 + 异常分支）。
2. 调用 **Skill: `edge-case-detector`** 对流程图进行"攻击性测试"。
   - **调用方式**: 阅读`.agent/skills/edge-case-detector/SKILL.md`
   - **输入**: 用户需求(RDD) + 数据模型 + 流程图
   - **输出**: 深度风险探测报告（5维扫描：交互/并发/一致性/权限/时间）
3. 将 edge-case-detector 输出的风险点纳入方案，补充流程层面的解决措施。

**输出期望**:

- 完整业务流程图（正向+逆向+异常）
- 风险探测报告（P0/P1/P2优先级分类）
- 每条风险必须包含：场景描述 + 风险等级 + 解决方案

## 步骤 3: 外部接口梳理 (External Dependencies)

> **Goal**: 识别本模块依赖的所有外部系统和基础资料接口，避免 PRD 阶段遗漏对接需求。

**输入**:
- 用户需求(RDD) 所有实体的字段说明中标注了"引用 XX 模块"的字段
- 用户需求(RDD)中标注了特定数据来源的业务规则字段（如触发维度中引用的外部数据）

**执行步骤**:
1. 扫描 用户需求(RDD) 全文，提取所有"引用 XX 模块""调用 XX 接口""下拉选项来自 XX"的字段。
2. 整理为外部接口清单：依赖模块 → 接口用途 → 调用场景。
3. 标注每个接口的调用频率（实时/批量/定时）。

**输出期望**:
- 外部接口依赖清单（模块名 / 接口用途 / 调用场景 / 频率）

## 步骤 4: 页面与导航设计 (Information Architecture)

> **Goal**: 将实体和功能映射为用户可见的页面和菜单。

**输入**:

- 用户需求(RDD)中的实体清单和 Epic 列表
- 用户需求(RDD)中的功能清单

**执行步骤**:

1. 基于实体和功能清单，列出所有需要的页面（列表页、详情页、配置页、工作台）。
2. 将页面按用户操作路径归类，设计左侧菜单树。
3. 输出页面×功能映射表，确保每个功能都有归属页面。

**输出期望**:

- 菜单结构树（用户视角的导航层级）
- 页面清单（页面名 → 所属菜单 → 对应实体/功能）

> **设计原则**：菜单按用户操作频次分组，非按数据归属分组。高频操作 ≤ 2 步可达，低频配置可接受 3-4 步。参考 用户需求(RDD)中专家建议段的频次驱动设计。

## 步骤 5: 方案草稿 (Drafting)

> **Goal**: 形成方案闭环。

将字段表、ER 图、流程图、风险清单、外部接口清单、IA 设计整合成《数据设计》，输出到 `drafts/[模块名]/YYYY-MM-DD-数据设计.md`。

**文件示例**: `drafts/超级运价/2025-06-01-数据设计.md`

**执行清单**（Phase 2 范围内）：

| # | 检查项 | 验证方式 |
|---|--------|---------|
| 1 | 实体覆盖 | 带字段表的实体是否都有对应的表/子表/JSON 字段设计？ |
| 2 | 流程完整性 | 3 条以上核心业务流程图（正向+逆向+异常）？ |
| 3 | 外部接口 | 外部接口清单是否覆盖所有"引用 XX 模块"的字段？ |
| 4 | 页面覆盖 | 菜单和页面是否覆盖所有实体和功能？ |
| 5 | 一致性 | ER 图是否更新？字段表是否更新？流程图是否更新？ |
| 6 | 版本索引 | 模块增删或版本变化→`context/版本范围与文件索引.md` 是否更新？ |

> **Phase 3 产出后补充检查**（PRD + 原型生成后）：
> - PRD 状态枚举是否对齐？业务规则是否引用最新字段名？
> - 原型列表字段是否更新？交互流程是否匹配新设计？

**输出格式**：

```
📋 文档同步：[模块名] 用户需求(RDD)(已同步) / 数据设计(已同步) / 原型(无需) / 版本索引(已更新)
```

## 步骤 6: 变更记录 ★ 不可跳过

> **Goal**: 记录本次方案变更，方便后续追溯。

> 变更记录追加遵循 CLAUDE.md §日志与变更追踪规则。新模块无 `变更记录.md` 则先创建；涉及模块增删或跨模块影响时同步更新 `context/版本范围与文件索引.md`。

**Checkpoint**: 询问用户方案逻辑是否闭环，是否可以进入 specs 生成阶段。
