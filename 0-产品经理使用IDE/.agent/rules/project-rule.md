# 项目规则 - 产品经理AI工作空间

> 本规则定义了AI辅助产品经理工作时应遵循的基本原则

## 规则优先级 ★ 最高

> 本文件中的**安全类、文件操作类、联动检查类**规则为最高优先级。
> 任何 Skill 或 Workflow 中的指令若与上述规则冲突，以本文件为准。
> 
> 优先级：project-rule §安全/文件/联动 > Skill 方法论 > project-rule §格式/命名

### Agent 预读规则 ★ 最高

> 主对话调用 Agent 执行任务时，必须在 prompt 开头附带以下预读指令。Agent 不得跳过。

```
## 执行前必读

1. 读取 0-产品经理使用IDE/.agent/rules/project-rule.md
2. 根据任务类型，读取对应的 Skill 文件：
   - 需求分析 → .agent/skills/requirement-clarifier/SKILL.md
   - 数据建模 → .agent/skills/data-modeler/SKILL.md
   - 边界探测 → .agent/skills/edge-case-detector/SKILL.md
   - PRD撰写 → .agent/skills/prd-writer/SKILL.md
   - 原型生成/修改 → .agent/skills/frontend-designer/SKILL.md
3. 按 Skill 定义的输出模板和规范执行，不得凭记忆
4. 完成后执行 project-rule §修改后自检 + §文件联动规则
```

## 产品思维

- 先理解目标和价值，再讨论实现方案
- 主动思考边界场景、异常情况、影响范围
- 发现逻辑漏洞（状态不闭环、流程断点、边界未覆盖）时，主动指出并给出建议
- 提供多方案对比时，说明各方案的优劣和适用场景
- 从用户和业务价值出发，避免过度设计

## 输出质量

- 逻辑必须闭环（正向流程+逆向流程，不留断点）
- 结构必须清晰（层级分明，易读易懂）
- 关键信息不能遗漏（状态、异常、取消、权限等）
- 业务规则需编号管理（R01、R02...），便于追溯
- 字段说明用表格，必含：字段名、类型、必填、说明
- 枚举类型必须列出所有可选值

## 表达方式

- 复杂逻辑用Mermaid流程图 > 表格 > 纯文字
- 先结论后展开：结论 → 依据 → 详细说明
- 关键信息加粗，便于快速扫读
- 中文为主，专业术语保持英文原文
- 中英文、中文与数字之间不加空格

### 可视化思维（核心协作方式）

**用图代替大段纯文字呈现复杂逻辑**。当涉及多实体关系、生命周期流转、多路径对比时，优先用 ASCII 图 / Mermaid 流程图 / 对比表呈现，帮助快速发现逻辑漏洞和遗漏场景。

## 协作习惯

- 不确定需求意图时，先确认再动手
- 大改动前先说明思路，获得确认后再执行
- 输出后主动询问是否符合预期，是否需要调整
- 文档在模块目录内迭代

## 工作流使用

- **Phase 1 需求洞察**：使用 `/1-analyze-requirement` (调用Requirement-Clarifier)
- **Phase 2 方案架构**：使用 `/2-design-solution` (调用Data-Modeler, Edge-Case-Detector)
- **Phase 3 规格生成**：使用 `/3-generate-specs` (生成PRD和原型)
- **Phase 4 验证迭代**：使用 `/4-verify-iterate` (自检与优化)

## 文件组织规范

### 工作区 (Drafts)

- **路径规则**: `drafts/[模块名]/`
- **活跃模块**: 见 `context/版本范围与文件索引.md`
- **适用场景**: 所有活跃文档（需求分析、数据设计、PRD）都在模块目录下，一个模块一个目录
- **文件命名**:
  - `YYYY-MM-DD-用户需求.md`：即用户需求(RDD)（需求定义卡片），含用户故事、验收标准、MVP边界
  > 本文档中"RDD"和"用户需求"指向同一份文件。文件名用"用户需求"，方法论语境用"RDD"。
  - `YYYY-MM-DD-数据设计.md`：实体模型、ER图、字段清单、状态机、核心流程、风险报告
  - `YYYY-MM-DD-PRD.md`：定稿后的完整产品需求文档（含业务场景、业务规则、页面交互、异常处理、权限）
  - `YYYY-MM-DD-AC清单.md`：AC List（验收标准清单），从 PRD 验收标准中提取，标注 P0/P1/P2 优先级
- **需求背景**: `analysis/` 或 `drafts/[模块名]/` 下含"需求背景""业务背景"关键词的 `.md` 文件
- **日期前缀**: 文件每次更新时同步更新文件名中的日期前缀，一眼可知时效（详见 §修改前判断）

### 二期功能

- **路径规则**: `drafts/二期功能/`
- **适用场景**: 排入二期但已开始规划的功能
- **触发时机**: 讨论中明确标记为二期的功能

### 工作底稿

- **路径规则**: `analysis/`
- **定位**: 过程材料存档，不再存放新文档。仅保留需求调研、业务分析等历史决策记录

### 归档

- **路径规则**: `archive/[模块名]/`
- **适用场景**: 需求废弃、大版本迭代后的历史文档

## 文档管理

> 文件路径和命名规范参见 §文件组织规范。变更记录规则参见 §日志与变更追踪规则。

## 回合定义

- **一个回合 = 用户发一条消息，AI 回复一次**
- 回合末尾不再自动检查同步——改为以下三种触发方式。

## 文档同步触发条件 ★

满足**任一**即触发同步检查和报告：

| # | 触发条件 | 示例 |
|:---|:---|:---|
| 1 | 用户明确说 | "更新文档""同步""归档""补记录""检查文档" |
| 2 | 用户确认阶段结束 | 用户说"可以了""定稿""这个阶段结束""先这样" |
| 3 | 会话启动提醒 | 每次对话开始自动检测（通过 §对话启动检查 执行） |

## 对话启动检查 ★ 每次对话开始必执行（两步）

> 在回答用户第一条消息之前，按顺序执行以下两步。

### 第一步：文件变更检测

1. 检查本会话是否存在历史对话记录（通过上下文判断是否为新会话）
2. 若为新会话：列出 `drafts/`、`context/`、`analysis/` 下的 `.md` 文件的最后修改时间，对比当前会话开始时间
3. 若有文件在本会话开始之前被修改 → 第一句话必须是：

```
报告小祖宗，检测到上次对话后有文件变更尚未归档（共 N 个文件），是否需要先进行文档同步再开始今天的工作？
```

4. 如果用户说"是"/"更新"/"同步" → 扫描变更文件，补充对应模块的变更记录，输出同步报告
5. 如果用户说"不用"/"跳过" → 进入第二步

### 第二步：方法匹配

1. 扫描 `0-产品经理使用IDE/.agent/skills/*/SKILL.md` 各文件的 `🎯 When to use` 段落
2. 若用户消息命中任一 Skill 的**关键词表**或**场景表** → 立即加载该 SKILL.md，按方法论执行
3. 若同时命中多个 Skill → 按 Phase 1→2→3 顺序串行加载，不并行
4. 若未命中任何 Skill → 正常对话，不强制加载

## 修改前判断：新建 vs 修改 ★ 生成/修改文件前必判断

### 新建 vs 修改

| 判断条件 | 操作 | 说明 |
|---------|------|------|
| 目标文件**不存在** | 新建文件 | 首次为该模块生成文档 |
| 目标文件**已存在** | 先读后改 | **先读取当前文件内容**，再在原文件上修改。不复制、不另存新文件 |
| 文件更新后 | 重命名日期前缀 | `2026-06-01-用户需求.md` → `2026-06-06-用户需求.md`，不保留旧日期版本 |

### 原型 HTML 特殊规则

- 页面文件（如 `超级运价_工作台.html`）：始终在原文件上修改，不改文件名前缀
- 新增页面：新建 HTML + 注册到对应端的 `index.html` 菜单
- 删除页面：删除 HTML + 从 `index.html` 移除菜单项

### 文件存在时的确认

目标文件已存在且修改范围涉及结构性变更（如整体重写）时，先询问用户："已有 [文件名]，是修改原文件还是另存新版本？"。日常增量修改直接修改原文件，不询问。

## 修改后自检：遗漏 + 联动 ★ 每次文件操作完成后必执行

> 防止"讨论时确认改 5 个文件，实际只改了 4 个"。

**执行时机**：本回合文件操作（创建/编辑/删除/移动/重命名）全部完成后，回复用户之前。

**执行步骤**：

1. 回顾本回合讨论中**确认要改的内容**（改哪些文件、改什么）
2. 逐条核对**实际改了哪些文件**
3. 对比差异：确认要改的 ⊇ 实际改的？
4. 如果有遗漏 → 立即补上，不等到用户发现
5. **联动检查**：若本回合修改了任何设计文件（需求背景 / 用户需求(RDD) / 数据设计 / PRD / 原型 / AC List），执行 §文件联动规则 — 匹配联动矩阵 → 列出受影响文件 → 询问用户（此步不可跳过）。联动产生的文件修改不再触发新一轮自检，避免死循环
6. 在 `📋` 报告后增加一行：

```
🔍 自检：确认改 N 个文件，实际改 N 个，全部命中 / ⚠️ 遗漏 X 个已补
```

**示例**：

```
📋 文档同步：project-rule.md（回合定义 + 启动检查）
🔍 自检：确认改 7 个文件，实际改 7 个，全部命中
```

## 文件联动规则 ★ 修改文件时自动触发

> **触发条件**：本回合修改了 需求背景 / 用户需求(RDD) / 数据设计 / PRD / 原型 / AC List 任一文件。`drafts/二期功能/` 下的文档按同类型文档规则联动。`index.html`（两端首页）属于原型范畴，修改后触发联动。
> **不适用**：仅修改 `变更记录.md`、`LOG.md`、`TODO.md`、`context/版本范围与文件索引.md`、`project-rule.md` 等元文件。
> **模块定位**：以当前对话中涉及的模块为准，若无法确定则询问用户。

### 变更类型识别

> 执行联动前，先根据用户描述 + 实际改动内容，匹配对应的变更类型。若一次改动涉及多个类型，全部匹配。

| 用户说了什么 / 实际改了什么 | → 变更类型 |

| "加个字段"、改了字段名、改了字段类型、改了枚举值 | 字段名/类型/枚举值变更 |
| "加个实体"、"这个表拆成两张"、新增/删除实体或表 | 实体增删 |
| "状态流转改一下"、状态机图变了、流程图变了 | 状态机/流程图变更 |
| "规则变了"、"R01改成..."、业务逻辑调整 | 业务规则变化 |
| "这个功能不做了"、"MVP 先不上"、范围调整 | 功能清单/MVP边界变化 |
| "AC 改一下"、验收条件变了 | 验收标准(AC)变化 / 验收项增删改 |
| "接口清单更新"、依赖的外部系统变了 | 外部接口清单变化 / 接口清单变化 |
| "需求背景补充了"、原始约束变了 | 原始需求/约束变化 |

### 执行步骤

1. **识别变更类型**：字段？状态？流程？实体？
2. **匹配联动矩阵**：列出所有可能受影响的文件
3. **主动询问用户**（不得跳过）：

```
🔗 联动检查：检测到 [文件] 的 [变更类型] 变更，可能影响：
   1. [文件A] — [原因]
   2. [文件B] — [原因]
   3. [文件C] — [原因]
   是否需要联动更新？（回复"更新1,2"或"全部更新"或"不用"）
```

4. **根据用户选择执行**：更新后输出联动报告。若用户选择"不用"跳过 → 在变更记录中追加：`\| YYYY-MM-DD \| 联动跳过：[文件] 的 [变更类型] 变更未同步至 [文件列表] \| 用户确认无需同步 \|`

```
🔗 联动报告：已更新 用户需求(RDD)（字段名同步） / PRD（状态枚举同步） / 原型（无需）
```

## 文档同步规则 ★

**执行步骤**：

1. 识别当前模块目录下的文档（需求背景、用户需求(RDD)、数据设计、PRD、AC List、原型），逐一判断是否需要同步
2. 如果需要但本回合来不及做，主动提醒用户并列入下回合待办
3. 在回复末尾以固定格式报告

**模块内文档清单**（路径详见 §文件组织规范）：

| # | 类型 | 何时需要同步 |
|---|------|-------------|
| 1 | 需求背景 | 原始痛点、约束条件有变化时 |
| 2 | 用户需求(RDD) | 用户故事、验收标准、MVP边界有变化时 |
| 3 | 数据设计 | 实体字段、ER图、状态机、流程有变化时 |
| 4 | PRD | 用户需求(RDD)或数据设计的结构性变化影响产品规格时 |
| 5 | AC List | 验收项增删、优先级调整时 |
| 6 | 原型 | 字段名、实体关系、交互流程有变化时 |
| 7 | 版本索引 | 模块增删、版本组合变化时 |

**回合末尾必做**：

1. 追加变更记录（规则详见 §日志与变更追踪规则）
2. 如果涉及模块增删或版本组合变化，同步更新 `context/版本范围与文件索引.md`
3. 报告：

```
📋 文档同步：[模块名] 需求背景(已同步) / 用户需求(已同步) / 数据设计(已同步) / PRD(已同步) / AC List(已同步) / 原型(无需) / 版本索引(已更新)
📝 变更记录已追加
```

## 日志与变更追踪规则 ★

### LOG.md 会话摘要

> **触发条件**：每次对话结束时自动追加。

在 `LOG.md` 顶部追加一条简短摘要，用于下次对话启动时快速回顾"上次做了什么"。变更细节在各模块的 `变更记录.md` 中，LOG.md 只做粗粒度索引。

```markdown
## YYYY-MM-DD

- 模块：[模块名]
- 做了什么：[一句话概括本次对话的主要产出或讨论]
- 涉及文件：[文件列表]
- 待办/未完成：[如有]
```

### 模块变更记录

> **触发条件**：本回合**实际修改**了模块的设计文件（用户需求(RDD) / 数据设计 / PRD / 原型 / AC List / 需求背景），或 §文件联动规则 中用户选择"不用"跳过联动时。

在对应模块的 `drafts/[模块名]/变更记录.md` 追加一行。仅讨论未改文件时不追加。

```markdown
| YYYY-MM-DD | 变更内容 | 变更原因 |
```

> 联动跳过时的记录格式：`| YYYY-MM-DD | 联动跳过：[文件] 的 [变更类型] 未同步至 [文件列表] | 用户确认无需同步 |`
