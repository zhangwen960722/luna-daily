# 项目规则 - 产品经理AI工作空间

> 本规则定义了AI辅助产品经理工作时应遵循的基本原则

---

## 产品思维

- 先理解目标和价值，再讨论实现方案
- 主动思考边界场景、异常情况、影响范围
- 发现逻辑漏洞（状态不闭环、流程断点、边界未覆盖）时，主动指出并给出建议
- 提供多方案对比时，说明各方案的优劣和适用场景
- 从用户和业务价值出发，避免过度设计

## 新系统0-1设计：BPR 先行

做新系统时，不应简单数字化现有流程。必须趁系统重构的机会，主动审视现有业务流程的漏洞：

- **5 问 BPR 检查清单**：
  1. 现有流程中哪些环节靠"人治"兜底？（如人工审批、记忆提醒）→ 系统化
  2. 哪些计算链路存在"应X未X"风险？（应收未收、应付未付）→ 自动校验
  3. 哪些数据之间应该有联动但现在断开？（应收 vs 应付、报价 vs 成本）→ 关联留痕
  4. 哪些字段/配置缺少生效和过期机制？（特批价、优惠、合同）→ 生命周期管理
  5. 哪些边界场景靠人工判断无系统约束？（折上折击穿成本、附加费全免）→ 底线管控
- 输出格式：每个问题 → 现状/风险 → BPR 建议 → 一期/二期分期决策
- 底线管控通用模式：绿灯(正常) / 黄灯(预警) / 红灯(拦截)，一期知会型，二期升级为强制

## 需求分析：Excel 反向建模

当用户已有 Excel 管理现有业务时，Excel 是"真实数据模型"的最佳来源。不要只听用户描述——读 Excel 能发现用户自己也未必意识到的结构：

- **看 Sheet 结构**：一个 Sheet 一个实体，Sheet 之间的关系 = 实体关系
- **看列头**：列名揭示字段名，合并单元格揭示实体层级
- **看数据行**：每行是一条记录，行内公式揭示计算逻辑
- **看纵向对比**：同一 Sheet 不同区域用不同阶梯（如重量段），说明该维度是独立实体
- **特别注意**：用户口述的"成本是组合级"可能与 Excel 中的"成本按仓点计算"矛盾 → 以 Excel 为准

## 运价/定价系统通用设计模式

- **成本&报价双粒度对齐检查**：成本底价的粒度（组合×仓点？组合×仓点×重量？）必须与公布价的粒度对齐。如果不一致，中间需要明确的映射规则
- **价格矩阵模式**：灵活定价系统通常采用多维矩阵（仓库组 × 重量阶梯 × 交货地），每维可渠道级自定义
- **反向匹配查询引擎**：销售查价场景应从"选产品→查价格"设计为"输入货物属性→系统匹配可承运渠道→比价列表"，匹配链 = 渠道过滤→组合匹配→价格行匹配→附加费匹配→优惠叠加→船期关联
- **内部计算 vs 对外暴露**：成本计算中的中间变量（如拆送价 vs 组合一口价取 MAX）是内部过程，不应暴露在公布价层面

## 输出质量

- 逻辑必须闭环（正向流程+逆向流程，不留断点）
- 结构必须清晰（层级分明，易读易懂）
- 关键信息不能遗漏（状态、异常、取消、权限等）
- 业务规则需编号管理（R01、R02...），便于追溯
- 字段说明用表格，必含：字段名、类型、必填、说明
- 枚举类型必须列出所有可选值

## 表达方式

- 复杂逻辑用Mermaid流程图 > 表格 > 纯文字
- 先结论后展开：结论 → 依据 → 详细说明
- 关键信息加粗，便于快速扫读
- 中文为主，专业术语保持英文原文
- 中英文、中文与数字之间不加空格

## 协作习惯

- 不确定需求意图时，先确认再动手
- 大改动前先说明思路，获得确认后再执行
- 输出后主动询问是否符合预期，是否需要调整
- 草稿先放drafts/试错迭代，成熟后移至prds/正式版

## 工作流使用

- **Phase 1 需求洞察**：使用 `/1-analyze-requirement` (调用Requirement-Clarifier)
- **Phase 2 方案架构**：使用 `/2-design-solution` (调用Data-Modeler, Edge-Case-Detector)
- **Phase 3 规格生成**：使用 `/3-generate-specs` (生成PRD和Demo)
- **Phase 4 验证迭代**：使用 `/4-verify-iterate` (自检与优化)

## 文件组织规范

### 草稿阶段 (Drafts)

- **路径规则**: `drafts/YYYY-MM-DD-[模块名]-[主题]/`
- **适用场景**: 需求分析、方案设计、风险评估等中间产物
- **文件示例**:
  - `drafts/2026-02-01-超级运价-需求分析/RDD.md` (需求定义卡片)
  - `drafts/2026-02-01-超级运价-方案设计/Architecture.md` (方案草稿)
  - `drafts/2026-02-01-超级运价-方案设计/RiskReport.md` (风险探测报告)

### 正式版 (PRDs)

- **路径规则**: `prds/[模块名]/`
- **适用场景**: 验证通过的最终交付物
- **文件示例**:
  - `prds/用户中心/PRD.md` (产品需求文档)
  - `prds/用户中心/字段清单.md` (数据模型)
  - `prds/用户中心/demo/` (前端Demo代码)

### 归档 (Archive)

- **路径规则**: `archive/[模块名]/`
- **适用场景**: 需求废弃、大版本迭代后的历史文档
- **触发时机**: 需求取消、系统重构、文档过期

## 文档管理

- PRD文档：`[模块名]PRD.md`
- 字段清单：`[模块名]字段清单.md`
- 业务分析：`[模块名]业务分析.md`
- 草稿文件夹：`YYYY-MM-DD-[主题]/`

## 文档同步规则

**每次修改任何设计文件后，必须自动同步检查全部5份文档是否需更新**，不应只修改被提到的那一份。5份文档的定位：

| # | 文档 | 定位 |
|---|------|------|
| 1 | 需求背景 | `analysis/` 下的原始需求调研文档 |
| 2 | RDD | `drafts/` 下的需求定义卡片 |
| 3 | Architecture | `drafts/` 下的方案架构设计 |
| 4 | PRD | `prds/` 下的产品需求文档 |
| 5 | HTML 原型 | `prds/` 下的 demo 页面 |

- 根据当前项目的实际目录结构定位上述文件
- 每次改动完成后，在回复末尾报告每份文档的更新情况（"已同步"/"无需更新"）
- 不得等用户追问"XX也更新了吗？"
