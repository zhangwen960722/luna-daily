# 🎯 需求定义卡片 (RDD) — 超级运价模块

> **原始需求**：从0-1搭建TMS跨境物流系统，超级运价是第一块核心模块。当前外采新智慧TMS等同于"线上记事本"，无法做成本计算、利润分析、运价管理。
>
> **文档版本**: v3.2 | **日期**: 2026-05-26 | **作者**: AI PM + 飞点业务团队

---

## 1. 核心洞察 (Insight)

### 真实痛点

飞点跨境物流作为亚马逊 FIST 承运商，当前的**核心瓶颈不是"没有系统"，而是"成本→报价"这条链路的计算全部依赖人工 Excel**。140个服务组合、每个组合 5 段成本、每段 3-5 个成本细项，每次海运费变动（几乎每周一次），运价管理员需要逐一手工换算 KG/CBM 成本价，一个组合耗时约 10 分钟，且存在汇率换算错误、公式引用错误的风险。

更深层的痛点是：**成本底价不可追溯**。当老板质疑"为什么这个报价这么低"时，没人能快速说清楚原始单价从哪里来、汇率是多少、公式怎么算的。

### JTBD（待办任务）

> **运价管理员**雇佣"超级运价"不是为了"管理成本"，而是为了：**当船公司周一早上发来新的海运费时，能在 30 分钟内完成全部 140+ 个服务组合的成本更新和对客公布价刷新，且每一分钱的变动都可追溯。**

> **销售**雇佣"超级运价"不是为了"查价格"，而是为了：**在客户微信群问"ONT8 这周什么价"时，5 秒内给出包含所有优惠的最终报价，不用切到 Excel 翻找，不用问运价管理员确认。**

### 业务价值：**HIGH**

| 维度 | 当前 | 目标 |
|------|------|------|
| 单组合成本核算 | 10 分钟（手工 Excel） | 1 分钟（系统自动计算） |
| 全量运价更新 | 1 天（逐一修改） | 0.5 天（批量更新） |
| 成本可追溯性 | 无（Excel 覆盖即丢失） | 100%（操作审计日志完整留痕） |
| 报价一致性 | 低（销售各自 Excel） | 高（统一公布价 + 优惠规则） |

## 2. 用户故事 (User Story)

### 服务渠道 (ServiceChannel)

服务渠道是超级运价模块的**最基础实体**。所有服务组合、成本、定价、优惠均挂在服务渠道之下。渠道配置是定价引擎运转的前提。

> **As a** 运价管理员
> **I want to** 创建一个服务渠道，定义其运输方式、尾程派送、包税方式等基本属性
> **So that** 后续的服务组合生成、成本核算、运价查询均有统一的渠道框架约束

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道名称 | 文本 | ✅ | 如"美森360" |
| 起运国 | 单选 | ✅ | 如"中国" |
| 目的国 | 多选 | ✅ | 如"美国""英国"，**控制 FBA 仓库下拉范围** |
| 运输方式 | 单选 | ✅ | 海运/空运/洲际火车/洲际卡车 |
| 尾程派送方式 | 多选 | ✅ | 卡派/快递派 |
| 包税方式 | 多选 | ✅ | 包税/不包税/自主税号/自税递延 |
| 计费方式 | 多选 | ✅ | 重量KG/体积m³ |
| 服务类型 | 多选 | ✅ | 散货/整柜 |
| 是否需要时效理赔 | 单选 | ✅ | Y/N |

> **约束**：尾程派送方式、计费方式创建后，该渠道下服务组合的对应选项受此约束。

#### B. 业务规则：目的国 × FBA 仓库过滤

FBA 仓库主数据需包含 `country` 字段（如 ONT8→美国，BHX4→英国）。服务渠道的目的国选择控制下游时效配置中的 FBA 仓库可选范围：

- 配置送仓预估时效/理赔承诺时效时，FBA 仓库下拉仅展示 `仓库.country ∈ 服务渠道.目的国` 的仓库
- 目的国减少时，如已有该国 FBA 仓库的时效配置，阻止操作并提示"需先清理该国仓库的时效配置后，方可移除此目的国"

---

### 预估运输时效 (EstimatedTransitTime)

挂在服务渠道上，**1:1** 关系。向亚马逊推送预计送达时段（Delivery Window），直接影响 FIST 考核的 DW 准确率。

> **As a** 运价管理员
> **I want to** 为服务渠道配置是否向亚马逊推送 FBA 运单 DW 及对应的服务等级和运输天数
> **So that** 亚马逊 FIST DW 考核有据可依，推送数据准确

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道 | channel_id | ✅ | |
| FBA运单同步 | Boolean | ✅ | 默认=否；控制是否向亚马逊推送 DW |
| 亚马逊服务等级 | Enum | 条件必填 | `标准` / `加急` / `快速`；FBA运单同步=是时必填 |
| 运输最小天数 | Integer | 条件必填 | 单位：天；FBA运单同步=是时必填 |
| 运输最大天数 | Integer | 条件必填 | 单位：天；FBA运单同步=是时必填，须 ≥ 运输最小天数 |

#### B. 业务规则

FBA运单同步=否时，另三个字段不存储（而非存储后 disabled）。如果未来切换为"是"，运价管理员必须主动重新填写——避免"过期数据静默激活"的风险。

---

### 送仓预估时效 (WarehouseDeliveryEstimate)

挂在服务渠道上，**1:N**（每个 FBA 仓库一条记录）。分两阶段向亚马逊更新送仓预估，提高 DW 准确率。

> **As a** 运价管理员
> **I want to** 为每个 FBA 仓库配置首次预估时效和提柜后的二次修正时效
> **So that** DW 分两阶段滚动更新，提柜后修正为更精确的送达时间

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道 | channel_id | ✅ | |
| FBA仓库代码 | String | ✅ | 受目的国过滤，下拉仅展示 `仓库.country ∈ 服务渠道.目的国` 的仓库 |
| 预估送仓时效（天） | Integer | ✅ | 第一阶段：运单创建时，DW = 开船日 + 此值，首次推送给亚马逊 |
| 提柜至送仓（天） | Integer | ✅ | 第二阶段：提柜后，DW = 提柜日 + 此值，重新预估并更新至亚马逊 |

#### B. 业务规则：DW 滚动更新

```
┌─────────────────────────────────────────────────┐
│  第一阶段（运单创建时）：                          │
│    DW = 开船日 + 预估送仓时效（天）               │
│    → 首次推送给亚马逊                              │
│                                                   │
│  第二阶段（运单提柜后）：                          │
│    DW = 提柜日 + 提柜至送仓（天）                  │
│    → 重新预估并更新至亚马逊                        │
└─────────────────────────────────────────────────┘
```

> **业务价值**：首次预估偏保守（用"预估送仓时效"），提柜后有了精确的提柜时间，用"提柜至送仓"做二次修正。两阶段滚动更新能显著提升 DW 准确率——这是 FIST Elite 承运商的核心考核指标。

#### C. 校验规则

提柜至送仓 ≤ 预估送仓时效，否则阻止保存并提示"提柜至送仓天数不可超过预估送仓时效"。

---

### 理赔承诺时效 (ClaimCommitment)

挂在服务渠道上，**1:N**（按派送方式 + FBA 仓库分组）。后续运单模块根据实际送达天数 vs 理赔承诺时效，自动判定是否需要理赔。

> **As a** 运价管理员
> **I want to** 按派送方式和 FBA 仓库配置理赔承诺时效
> **So that** 运单延误理赔有标准可查，系统自动判定触发条件

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道 | channel_id | ✅ | |
| 派送方式 | Enum | ✅ | `快递派` / `卡派`；从服务渠道的尾程派送方式中取值 |
| FBA仓库代码 | 多选 | 条件必填 | 卡派必填（不同仓点可配置不同理赔时效）；快递派非必填（快递网络覆盖全美，时效相对一致） |
| 理赔时效（工作日） | Integer | ✅ | 单位：工作日；实际送达天数 > 此值 → 触发理赔判定 |

#### B. 业务规则

- 服务渠道 `是否需要时效理赔` = N → 理赔承诺时效配置不可编辑，已有数据清空
- 卡派未选仓库 → 阻止保存并提示"卡派必须指定 FBA 仓库"
- 同一仓点不可重复出现在多条理赔规则中

### 港口 (Port)

港口数据属于基础数据模块，不在超级运价内维护。超级运价通过 `port_code` 引用。航线中的出发港/目的港字段存储的是港口代码，非中文名。

#### A. 字段定义

| 字段 | 类型 | 说明 |
|------|------|------|
| 港口名称 | String | 如"上海港"、"长滩港" |
| 港口代码 | String | 唯一标识，如 `USLAX`、`CNSGH` |
| 国家 | String | 如"中国"、"美国" |
| 类型 | Enum | 出发港 / 目的港 |

---

### 航线 (Route)

航线是连接"报价端"和"亚马逊推送端"的关键桥梁——提单绑定航线，运单绑定提单，运单推送到亚马逊 FIST 时从提单→航线获取港口代码。

> **As a** 系统
> **I want to** 运单推送亚马逊 FIST 时，从提单关联的航线中自动获取出发港代码和目的港代码
> **So that** 关键事件节点（干线出发/到达）的港口信息准确且格式符合亚马逊要求

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 航线代码 | String | | 船司分配的航线代码，如 `AAC/AAC2`、`CLX`、`SEA/SEA3`。默认为空，可编辑。同一船司的同一港口对可能有多条航线代码（如 COSCO 宁波→长滩同时有 AAC/AAC2 和 AAC/AAC4/PS5），因此不能用出发港+目的港拼接 |
| 出发港 | port_code | ✅ | 引用港口主数据，展示格式 `港口名称（港口代码）`如"长滩港（USLAX）"，数据库存名称+代码 |
| 目的港 | port_code | ✅ | 同上 |
| 船司 | supplier_id | ✅ | 引用 SRM 供应商管理模块（供应商大类"干线运输供应商"），不在超级运价内重复维护 |
| 预计运输天数 | Integer | | 配置在航线级别，船期可覆盖 |

#### B. 关联链路

航线 N:N 关联服务渠道（用于报价展示），1:N 关联提单（运单推送 FIST 时自动获取港口代码）。

```
港口主数据 ──→ 航线 ──→ 服务渠道（N:N，用于报价展示）
              │
              └──→ 提单（1:N，提单绑定一条航线）
                     │
                     └──→ 运单（1:N，运单绑定一张提单）
                            │
                            └──→ Amazon FIST 推送
                                   ├── 干线出发事件 → 取 航线.出发港代码
                                   └── 干线到达事件 → 取 航线.目的港代码
```

> 一条渠道可配多条航线（如快船600同时有 SEA/SEA3 和 AAC/AAC2 等航线）。一个提单只绑定一条航线。Amazon FIST 推送时，运单通过提单→航线直接拿到港口代码，无需在运单层面重复维护港口信息。

### 周船期批次 (WeeklyScheduleBatch)

运价管理员的工作粒度是"周"——每周一早上同步更新所有航线的本周船期和运价。周船期批次是按周汇总的主列表，作为「运价更新工作台」的核心入口。

> **As a** 运价管理员
> **I want to** 按周创建船期批次，系统自动管理批次状态
> **So that** 每周的船期和运价属于同一运维周期，查询时只展示当前周数据

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 批次名称 | String | ✅ | 如 "2026-W05（1/26-2/1）" |
| 周次标识 | String | ✅ | ISO 周次，如 `2026-W05`，唯一索引 |
| 周起始日 | Date | ✅ | 周一日期 |
| 周结束日 | Date | ✅ | 周日日期 |
| 状态 | Enum | ✅ | PREVIEW（预告）/ CURRENT（当前）/ EXPIRED（已过期） |
| 备注 | String | | |

#### B. 业务规则：状态流转

每周日 24:00 系统自动将 CURRENT → EXPIRED，下周的 PREVIEW → CURRENT。同一时刻只有一条 CURRENT 批次。运价查询时只展示 CURRENT 周的船期。

**运价更新工作台 — 周船期主列表**：

```
┌───────────────────────────────────────────────────────────────────────────┐
│  运价更新工作台 — 本周船期（2026-W05，1/26-2/1）                           │
│                                                                           │
│  航线                  截单(广州) 截单(汕头) ETD    ETA    航程 最快提取   │
│  ──────────────────── ──────── ──────── ────── ────── ──── ────────     │

│  MAX 上海→长滩(MATSON)  2月2日   2月2日   2月6日 2月17日 11天  2月18日    │
│  SEA/SEA3 盐田→长滩    1月31日  1月31日  2月4日 2月15日 11天  2月16日    │
│  ...                   ...      ...      ...    ...     ...   ...        │
│                                                                           │
│  [新增航线船期]  [从上周复制]  [批量导入]                                   │
└───────────────────────────────────────────────────────────────────────────┘
```

### 船期 (SailingSchedule)

每条船期记录挂在周批次下，关联一条航线。船期表是运价查询时销售必看的信息——"这周有没有船？什么时候开？什么时候到？"

> **As a** 销售
> **I want to** 在运价查询结果中看到每个渠道的船期信息（船司、航线、ETD、ETA）
> **So that** 客户问"我的货什么时候能到"时，当场给出准确答复

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 周批次 | batch_id | ✅ | 所属周批次 |
| 航线 | route_id | ✅ | 关联航线 |
| 截单时间 | DateTime | ✅ | 按本地仓库配置，每个仓库一个截单时间（如广州仓、汕头仓、义乌仓各自独立），仓库引用仓库列表的"仓库名称" |
| 预计开船时间 (ETD) | Date | ✅ | |
| 预计到港时间 (ETA) | Date | ✅ | |
| 航程（自然日） | Integer | ✅ | ETD→ETA 天数 |
| 最快提取/送仓时间 | Date | | 开船后第 X 天 |

#### B. 业务规则：双视角展示

周船期主列表是"运维视角"（本周有哪些船），航线历史是"追溯视角"（这条航线过去/未来的船期变化）。运价更新工作台默认展示本周主列表，点击航线名称可展开该航线的历史船期。

**单航线船期历史**：

└──────────────┴──────────┴──────────┴──────────┴──────────┴──────┴──────────┘
```

> **维护频率**：每周维护，与运价更新同频。"运价更新工作台"首页即本周船期主列表，运价管理员在同一界面完成本周所有航线的船期录入和运价更新。

### 重量段配置 (WeightTier)

挂在服务渠道上，该渠道下所有组合共享。每个服务渠道可定义自己的重量段阶梯，不同渠道阶梯不同。

> **As a** 运价管理员
> **I want to** 为每个服务渠道自定义重量段阶梯
> **So that** 不同渠道（海运/空运/快递）可使用不同的重量分段规则

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道 | channel_id | ✅ | 挂在哪个服务渠道下 |
| 重量段名称 | String | ✅ | 如"8KG+"、"101KG+"、"47-72kg" |
| 最小重量 | Decimal | ✅ | 单位 KG，该段起始重量（含） |
| 最大重量 | Decimal | | 单位 KG，该段结束重量（不含）；null = 以上 |

示例：
```
美森360（海运卡派）:  8KG+ | 101KG+
美国空运240（空运）:   12KG+ | 21KG+ | 51KG+ | 101KG+
国际快递（UPS/DHL）:   23kg+ | 47-72kg | 73-99kg
```

### 运价行 (PriceTableRow)

公布价不是单一数字，而是**仓库组 × 重量段 × 交货地**的三维矩阵。每条运价行对应一个"价格 SKU"。

> **As a** 运价管理员
> **I want to** 为每个服务组合按仓库组 × 重量段 × 交货地维护公布价
> **So that** 系统能根据销售输入的货物参数精确匹配对应价格

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务组合 | combination_id | ✅ | |
| 仓库组 | String (逗号分隔) | ✅ | 如 `ONT8,LAX9,LGB8` |
| 重量段 | weight_tier_id | ✅ | 渠道级定义的重量段 |
| 交货地 | pickup_region | ✅ | 珠三角/汕头/义乌 |
| 公布价-per_KG | Decimal | 条件必填 | 若计费方式含 KG |
| 公布价-per_CBM | Decimal | 条件必填 | 若计费方式含 CBM |
| 参考签收时效 | String | | 如"12-16个自然日" |
| 理赔时效 | Integer | | 工作日，从 needClaim 配置继承可覆盖 |

#### B. 与成本底价的关系

```
成本底价(组合, 仓点A) = 头程共享 + 派送段(仓点A)
成本底价(组合, 仓点B) = 头程共享 + 派送段(仓点B)

运价行(组合, 仓库组="A,B,C", 重量段, 交货地) → 公布价
  └── 仓库组覆盖仓点 A、B、C，取三个仓点中成本最高的作为定价基准
  └── 利润 = 公布价 - max(成本底价A, 成本底价B, 成本底价C)
```

> **设计决策**：仓库组用逗号存而非独立表——当前分组稳定，15 个渠道约需 50-80 个仓库组，逗号方案简单够用。未来如果需要仓库组独立管理（如改名、移动仓点），再迁移至独立表。

### Epic 1：成本底价管理

> **As a** 运价管理员
> **I want to** 为每个服务组合维护 5 段成本细项，系统自动计算头程共享成本和派送段按仓点成本，汇总为每个仓点的最终成本底价
> **So that** 我不再用计算器手工换算，且成本底价变动有据可查

**成本模型（两层结构）**：

```
成本底价(组合, 仓点) = 头程共享成本(组合) + 派送段成本(组合, 仓点)

头程共享成本（4段，所有仓点共用）：
  揽收段 + 干线运输段 + 关务段 + 仓储段
  → 按 KG标准 / CBM标准 均摊为 per_KG 和 per_CBM 单价

派送段成本（1段，按仓点独立）：
  每个 FBA 仓点有三种供应商报价来源：
    ├── 拆送价（按每箱派送）
    ├── 组合一口价（按整板派送）
    └── 整柜直送价（仅整柜适用）
  → 取 MAX(拆送价, 组合一口价) 作为散货卡派的派送段成本底价
  → 拆送 vs 组合仅为内部计算过程，对外不暴露

最终成本底价 = 头程共享 per_KG + 派送段 per_KG（按仓点）
              └── 粒度：服务组合 × FBA仓点
```

#### 成本项 (CostItem)

每个成本段下包含若干成本细项，是成本底价计算的最小单元：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务组合 | combination_id | ✅ | |
| 成本段 | Enum | ✅ | `揽收段` / `干线运输段` / `关务段` / `仓储段` / `派送段` |
| 成本项名称 | String | ✅ | 如"海运费"、"报关费"、"拆送价" |
| 原始单价 | Decimal | ✅ | 供应商报价原始金额 |
| 币种 | Enum | ✅ | `RMB` / `USD` |
| 汇率 | Decimal | 条件必填 | 币种=USD 时必填，用于 USD_CONVERT 公式 |
| cost_per_kg | Decimal | | 系统自动计算 = 原始单价按 KG 标准均摊 |
| cost_per_cbm | Decimal | | 系统自动计算 = 原始单价按 CBM 标准均摊 |
| 交货地 | pickup_region | 条件必填 | 成本段=揽收段时必填，如"珠三角""汕头""义乌" |
| 派送报价来源 | Enum | 条件必填 | 成本段=派送段时必填：`拆送价` / `组合一口价` / `整柜直送价` |
| FBA仓库代码 | String | 条件必填 | 成本段=派送段时必填 |
| effective_from | DateTime | | 未来生效时间；一期预留字段，手动切换 |

> **计算公式**：详见 [4. 核心算法 — STEP 1](#4-核心算法最终报价计算引擎)。派送段取 MAX(拆送价, 组合一口价) 作为成本底价，整柜取直送价。

**核心场景**：
1. 创建服务渠道（如"美森360"）→ 选择"保存并生成服务组合" → 系统根据干线×尾程×收费×包税×服务类型自动生成组合，并全量继承 5 段全部成本细项（含默认币种和原始单价）
2. 运价管理员对每个组合做减法（移除不适用的成本项）+ 覆盖单价（如海运费从默认值覆盖为本周实际报价）
3. 系统根据公式自动计算每个成本项的 cost_per_kg 和 cost_per_cbm
4. 派送段独立录入多来源报价（组合一口价 / 拆送价按仓点 / 整柜直送价），成本底价取 MAX
5. 海运费变动时，运价管理员可选：(a) 在成本段管理更新单价 → 选择关联服务组合批量刷新，或 (b) 进入单个服务组合单独更新
6. **成本项支持"未来生效"**：修改成本单价时可指定 `effective_from`，系统在该时间点自动切换新价格。周五录入下周一的海运费，周一 00:00 自动生效，消除"更新窗口期间报价不准"的风险。一期仅做字段预留 + 手动切换，二期做定时自动切换。

### Epic 4：附加费规则引擎（配置端）

> **As a** 运价管理员
> **I want to** 灵活定义附加费规则（触发条件 + 计价公式 + 费用方向），系统在运单创建时自动匹配并计算

**核心场景**：

#### 4A. 规则创建

运价管理员创建一条附加费规则时，需要配置以下要素：

| 配置项 | 说明 | 示例 |
|--------|------|------|
| 规则名称 | 唯一标识 | "出口报关费（0110/9810）" |
| 费用类型标签 | 用于报表分类 | "出口费用" / "国内费用" |
| 费用方向 | 应收(+) / 客户优惠(-) | 报关费=应收；客户自送=客户优惠 |
| 可优惠级别 | FULL / PARTIAL / NONE | 报关费=NONE（政府代收）；贴标费=FULL |
| 触发方式 | AUTO / MANUAL / BOTH | 报关费=AUTO；贴标费=MANUAL |
| **触发条件** (AUTO/BOTH 时必填) | 货物属性 + 运单属性的组合条件 | 报关方式 IN {0110, 9810} |
| **计价规则** | 计价单位 + 单价 + 币种 | 160元/票 |
| 人工输入字段 (MANUAL/BOTH 时) | 操作员在运单界面填入的数量 | 标签="贴标张数"，类型=NUMBER |
| 生效范围 | GLOBAL / 指定渠道 / 指定组合 | 全局 |
| 状态 | ACTIVE / DISABLED | ACTIVE |

**触发条件支持的判断维度**：

```
货物属性条件：
  ├── 单件重量 > X KG
  ├── 单件长度 > X CM
  ├── 单件尺寸(长+宽+高) > X CM
  ├── 品名含关键词（如"粉末""液体"）
  └── 产品类型 = 特定分类

运单属性条件：
  ├── 报关方式 = 0110 / 9810 / ...
  ├── 自送标记 = 是
  ├── 投保标记 = 是
  ├── 目的仓点 ∈ 指定列表
  └── 服务渠道 ∈ 指定列表

条件组合：支持 AND / OR
```

**计价单位（6 种）**：

| 计价单位 | 计算公式 | 示例 |
|---------|---------|------|
| PER_TICKET | unit_price × 1 | 报关费 160元/票 |
| PER_BOX | unit_price × 箱数 | 超重费 200元/箱 |
| PER_KG | unit_price × 计费重量 | 粉末加收 1.5元/KG |
| PER_CBM | unit_price × 计费体积 | — |
| PER_UNIT | unit_price × 人工填入数量 | 贴标费 2元/张 |
| PER_DECLARED_VALUE% | declared_value × rate% | 保险费 0.3% |

#### 4B. 自动匹配流程

```
运单创建时：
  1. 查询所有 status=ACTIVE、触发方式∈{AUTO, BOTH} 的附加费规则
  2. 逐一匹配 trigger_condition 与当前运单的货物属性+运单属性
  3. 命中后按计价单位计算金额
  4. 汇总所有命中的附加费，生成业务流水记录
```

**案例还原（来自实际业务）**：

> 运单：美森360 → ONT8，3箱，1200KG，报关方式=0110，产品=粉末
>
> 系统自动命中：
> - 规则"粉末品名加收"：1.5元 × 1200KG = 1800元
> - 规则"出口报关费（0110/9810）"：160元 × 1票 = 160元
> - 规则"超重附加费"（单箱>30KG触发）：200元 × 3箱 = 600元
>
> 自动附加费合计 = 2560元
>
> 操作员在运单界面手动添加：
> - 规则"贴标费"：2元 × 3张 = 6元
>
> 附加费总计 = 2566元

#### 4C. 客户自送优惠（特殊规则）

```
规则名称："客户自送优惠"
费用方向：客户优惠(-)
触发方式：AUTO
触发条件：自送标记 = 是
计价单位：PER_KG
单价：-1元/KG

效果：当运单标记"自送"时，系统自动创建一条应收为负的业务流水
```

> **设计要点**：客户自送优惠本质上也是附加费规则，只是费用方向为"客户优惠(-)"。统一放在附加费规则引擎管理，不单独开模块。

|-------------------|------|---------|
| FULL | 服务型附加费（如贴标费） | 可全免，折扣率 0~∞ |
| PARTIAL | 成本型附加费（如粉末加收） | 最低收成本价，折扣率 ≥ 成本占比 |
| NONE | 代收代缴型（如政府税费） | 不可配置任何优惠 |

> **一期实现**：建表时预留 `discountable_level` 字段，创建优惠规则时做软校验（弹窗提示，不强制拦截）。二期完善硬拦截逻辑。

#### 6C. 附加费优惠与运费优惠的关系

```
最终报价 = (公布价 × Σ运费折扣率) + Σ(附加费金额 × Σ附加费折扣率)

两类优惠独立计算，互不影响（"折上折"仅指各自内部乘法叠加）：
  - 运费优惠的折扣不会影响附加费
  - 附加费优惠的折扣不会影响运费
```

**案例还原（来自实际业务）**：

> 客户：VIP 等级，产品粉末
> 美森360 ONT8 卡派，公布价 9元/KG
>
> 运费部分：
>   - VIP 等级折扣 9折 → 9 × 0.9 = 8.1元/KG
>
> 附加费部分：
>   - 粉末加收：+1.5元/KG
>   - VIP 附加费优惠：粉末附加费 -0.5元/KG（折扣率 0.67）
>   - 实际粉末附加费：1.5 × 0.67 = 1.0元/KG
>
> 最终报价 = 8.1 + 1.0 = 9.1元/KG

### Epic 7：运价查询与最终报价（销售视图 — 反向匹配引擎）

> **As a** 销售
> **I want to** 输入货物属性（产品类型、件数、重量、尺寸、目的国、目的仓），系统自动匹配所有符合条件且可承运的服务渠道及报价
> **So that** 客户说"3 件粉末，每件 60KG，发 ONT8"时，我当场就能对比美森360 vs 美森432 vs EX456 的多维度报价

#### 7A. 查询入口

```
销售输入（必须）：
  ├── 目的国          → 过滤服务渠道.目的国
  ├── 目的仓点        → 过滤送仓预估时效中有该仓点的渠道

  ├── 计费方式偏好    → KG / CBM（可多选）
  └── 产品类型（可选）→ 匹配附加费规则 + 产品限制

销售输入（可选）：
  ├── 交货地          → 匹配运价行的 pickup_region，默认珠三角
  └── 客户名称        → 匹配运费优惠 + 附加费优惠
```

#### 7B. 匹配流程

```
Step 1 — 渠道初筛
  Filter: 服务渠道.目的国 contains 输入的目的国
  Filter: 送仓预估时效 contains 输入的目的仓点
  Filter: 件数 ∈ [最小件数, 最大件数]
  → 候选渠道列表

Step 2 — 组合匹配
  对每个候选渠道，找到其所有 ACTIVE 服务组合
  Filter: 计费方式 match 输入偏好
  → 候选组合列表

Step 3 — 运价行匹配
  对每个候选组合：
    Filter: 仓库组 contains 输入的目的仓点
    Filter: 重量段 match 输入的总重量/材重
    Filter: 交货地 = 输入的交货地
    → 取匹配的运价行.公布价
    → 如未匹配（重量不在任何段内），标记"需人工报价"

Step 4 — 附加费匹配
  对命中的组合，匹配附加费规则：
    Filter: 产品类型触发附加费（如"粉末"→ +1.5元/KG）
    Filter: 货物属性触发（超重超长等）
    → 计算附加费明细

Step 5 — 优惠叠加（若选择了客户）
  匹配运费优惠 → 折上折乘法叠加
  匹配附加费优惠 → 独立计算

Step 6 — 船期关联
  对每个命中渠道，查询 CURRENT 周船期批次中的船期（ETD、ETA、航程）
  → 展示"下一班船：X 月 X 日出发，预计 X 月 X 日到港"

Step 7 — 底线检查
  成本底价 = 头程共享 + 派送段(该仓点)
  盈亏标识 = GREEN / YELLOW / RED
```

## 4. 核心算法：最终报价计算引擎

这是整个超级运价模块的心脏。以下伪代码定义了从成本底价到最终报价的完整计算链路。

```
┌─────────────────────────────────────────────────────────────┐
│  FUNCTION: calculateFinalQuote(                              │
│    service_combination_id,  // 服务组合                       │
│    warehouse_id,            // 目的仓点                        │
│    customer_id,             // 客户                           │
│    cargo_params             // 货物参数 {weight, volume, ...}  │
│  ) → FinalQuote                                             │
└─────────────────────────────────────────────────────────────┘

// ===== STEP 1: 成本底价（头程共享 + 派送按仓点） =====
// 头程共享成本（4段：揽收+干线+关务+仓储）—— 所有仓点共用
shared_cost_items = query_cost_items(service_combination_id, segment IN [揽收,干线,关务,仓储])
shared_per_kg  = SUM(shared_cost_items.map(ci → ci.cost_per_kg))
shared_per_cbm = SUM(shared_cost_items.map(ci → ci.cost_per_cbm))

// 派送段成本 —— 按仓点独立
delivery_items = query_cost_items(service_combination_id, segment = "派送段", warehouse = target_warehouse)
// delivery_items 包含三种报价来源：
//   拆送价、组合一口价、整柜直送价（仅整柜）
//   散货卡派取 MAX(拆送价, 组合一口价)；整柜取直送价
delivery_per_kg  = MAX(delivery_items.filter(applicable).map(d → d.cost_per_kg))
delivery_per_cbm = MAX(delivery_items.filter(applicable).map(d → d.cost_per_cbm))

// 最终成本底价 = 头程共享 + 派送段
cost_per_kg  = shared_per_kg  + delivery_per_kg
cost_per_cbm = shared_per_cbm + delivery_per_cbm


// ===== STEP 2: 公布价（矩阵匹配） =====
// 公布价不是公式计算，而是从运价表中按矩阵维度匹配
price_row = query_price_table(
    service_combination_id = combination_id,
    warehouse_group_contains = target_warehouse,   // 仓库组包含目标仓点
    weight_tier_matches = chargeable_weight,        // 重量段匹配
    pickup_region = input_pickup_region             // 交货地匹配
)

if price_row found:
    published_per_kg  = price_row.price_per_kg
    published_per_cbm = price_row.price_per_cbm
else:
    // 未匹配任何运价行 → 标记"需人工报价"
    return { status: "NO_MATCH", message: "该条件暂无报价，请联系运价管理员" }


// ===== STEP 3: 运费优惠（折上折） =====
customer_level = query_customer_level(customer_id)

freight_discounts = query_freight_discounts(
    customer_id    = customer_id,
    customer_level = customer_level,
    status         = ACTIVE,
    effective_at   = NOW()
)

// 折扣率乘法叠加
freight_discount_factor = 1.0
for each discount in freight_discounts.sorted_by_priority():
    freight_discount_factor = freight_discount_factor × discount.rate

discounted_per_kg  = published_per_kg  × freight_discount_factor
discounted_per_cbm = published_per_cbm × freight_discount_factor


// ===== STEP 4: 价格底线检查（BPR-1） =====
SAFETY_MARGIN = 1.05  // 安全边际系数

if discounted_per_kg >= cost_per_kg × SAFETY_MARGIN:
    profit_flag = GREEN
elif discounted_per_kg >= cost_per_kg:
    profit_flag = YELLOW    // 薄利
else:
    profit_flag = RED       // 亏本，系统弹窗提示


// ===== STEP 5: 附加费自动匹配 =====
surcharge_rules = query_surcharge_rules(
    status       = ACTIVE,
    trigger_mode ∈ {AUTO, BOTH},
    scope         = matching(service_combination_id)
)

matched_surcharges = []
for each rule in surcharge_rules:
    if match_conditions(rule.trigger_condition, cargo_params, waybill_attrs):
        amount = calc_surcharge(rule.pricing, cargo_params)
        matched_surcharges.append({
            rule_id:       rule.id,
            rule_name:     rule.name,
            fee_direction: rule.fee_direction,  // RECEIVABLE / CUSTOMER_DISCOUNT
            original_amount_per_kg:  amount.per_kg,
            original_amount_total:   amount.total,
            pricing_unit:  rule.pricing.unit
        })

// calc_surcharge 逻辑：
//   PER_TICKET          → unit_price × 1
//   PER_BOX             → unit_price × cargo_params.box_count
//   PER_KG              → unit_price × cargo_params.chargeable_weight
//   PER_CBM             → unit_price × cargo_params.chargeable_volume
//   PER_UNIT            → unit_price × manual_input.quantity (AUTO时不计算，等人工填)
//   PER_DECLARED_VALUE% → declared_value × unit_price%


// ===== STEP 6: 附加费优惠 =====
surcharge_discounts = query_surcharge_discounts(
    customer_id    = customer_id,
    customer_level = customer_level,
    status         = ACTIVE
)

for each surcharge in matched_surcharges:
    applicable_discount = find_best_match(surcharge.rule_id, surcharge_discounts)
    if applicable_discount:
        // 乘法叠加（折上折）
        surcharge_discount_factor = 1.0
        for each sd in applicable_discount:
            surcharge_discount_factor = surcharge_discount_factor × sd.rate
        
        surcharge.discount_rate       = surcharge_discount_factor
        surcharge.final_amount_per_kg = surcharge.original_amount_per_kg × surcharge_discount_factor
        surcharge.final_amount_total  = surcharge.original_amount_total  × surcharge_discount_factor
    else:
        surcharge.discount_rate       = 1.0
        surcharge.final_amount_per_kg = surcharge.original_amount_per_kg
        surcharge.final_amount_total  = surcharge.original_amount_total


// ===== STEP 7: 汇总最终报价 =====
total_surcharge_per_kg  = SUM(matched_surcharges.map(s → s.final_amount_per_kg))
total_surcharge_per_cbm = SUM(matched_surcharges.map(s → s.final_amount_per_cbm))

final_per_kg  = discounted_per_kg  + total_surcharge_per_kg
final_per_cbm = discounted_per_cbm + total_surcharge_per_cbm


// ===== STEP 8: 返回 =====
return FinalQuote {
    cost_base: {
        per_kg:  cost_per_kg,
        per_cbm: cost_per_cbm
    },
    published_price: {
        per_kg:  published_per_kg,
        per_cbm: published_per_cbm
    },
    freight_discount: {
        rules:          freight_discounts,
        factor:         freight_discount_factor,
        discounted_per_kg:  discounted_per_kg,
        discounted_per_cbm: discounted_per_cbm
    },
    surcharges:        matched_surcharges,  // 含 original + discount + final
    final_quote: {
        per_kg:  final_per_kg,
        per_cbm: final_per_cbm
    },
    profit_flag:       profit_flag,         // GREEN / YELLOW / RED
    safety_margin:     SAFETY_MARGIN
}
```

     → 本周船期主列表（WeekScheduleBatch，展示所有航线本周船期一览）
     → 看到"本周待更新成本项"列表（海运费标红）
     → 批量修改 → 展示受影响组合 → 勾选 → 一键重算
     → 同一页面中可切换"船期"和"运价"两个 Tab
```

**设计原则**：高频操作 ≤ 2 步，低频配置可接受 3-4 步。船期和运价在同一工作台维护，因为二者是同频的——船公司周一发来新海运费时，往往也同步更新船期。

### 建议 2：海运费更新用"成本段 → 勾选推送到关联组合"模式

修改海运费单价后，展示"受影响的服务组合清单"（列出当前公布价），让管理员**勾选后批量刷新**。不做无脑全量刷新——可能存在某些组合正在谈价，暂不想变动。

### 建议 3：特批价流程不做系统内审批

飞书审批链已成熟运行。系统只保留"录入特批价结果"功能，关联飞书审批编号即可。(1) 一期开发量可控；(2) 0-1 阶段不碰 BPM 工作流。

### 建议 4：EXCEL 派送段数据做"结构化清洗"

《美线头程成本核算表》中，一个仓点行包含 6 组数据（拆送/组合/直送 × KG/CBM）及末端/头程拆分。导入时引导运价管理员按仓点逐行录入，系统自动算 MAX，不直接套 Excel 格式。

### 建议 5：预留 `cost_segment_link` 表结构

一期全段服务默认全量 5 段关联，二期分段服务可灵活选择部分段关联。表结构一期建好——SQL 成本极低，二期改表成本极高。

### 建议 6：一期建好 BPR 字段，二期完善逻辑

以下字段一期建表时就要有，但业务逻辑二期再完善：
- `discountable_level`（附加费可优惠性）：一期软校验（弹窗提示），二期硬拦截
- `cost_item_id`（附加费应收-应付关联）：一期存 NULL，二期填数据做毛利率分析
- `profit_flag`（价格底线标识）：一期展示 + 弹窗确认，三期升级为硬拦截

### 建议 7：附加费规则引擎 — 先建"案例驱动"的预置规则，再开放自定义

一期上线时预置当前已知的规则（超重超长、报关费、粉末加收、贴标费、客户自送），确保开箱即用。自定义规则功能同期上线，但不作为上线验收的硬条件。
