# 🎯 需求定义卡片 (RDD) — 超级运价模块

> **原始需求**：从0-1搭建TMS跨境物流系统，超级运价是第一块核心模块。当前外采新智慧TMS等同于"线上记事本"，无法做成本计算、利润分析、运价管理。
>
> **文档版本**: v3.3 | **日期**: 2026-05-27 | **作者**: AI PM + 飞点业务团队

## 1. 核心洞察 (Insight)

### 真实痛点

飞点跨境物流作为亚马逊 FIST 承运商，当前的**核心瓶颈不是"没有系统"，而是"成本→报价"这条链路的计算全部依赖人工 Excel**。140个服务组合、每个组合 5 段成本、每段 3-5 个成本细项，每次海运费变动（几乎每周一次），运价管理员需要逐一手工换算 KG/CBM 成本价，一个组合耗时约 10 分钟，且存在汇率换算错误、公式引用错误的风险。

更深层的痛点是：**成本底价不可追溯**。当老板质疑"为什么这个报价这么低"时，没人能快速说清楚原始单价从哪里来、汇率是多少、公式怎么算的。

### JTBD（待办任务）

> **运价管理员**雇佣"超级运价"不是为了"管理成本"，而是为了：**当船公司周一早上发来新的海运费时，能在 30 分钟内完成全部 140+ 个服务组合的成本更新和对客公布价刷新，且每一分钱的变动都可追溯。**

> **销售**雇佣"超级运价"不是为了"查价格"，而是为了：**在客户微信群问"ONT8 这周什么价"时，5 秒内给出包含所有优惠的最终报价，不用切到 Excel 翻找，不用问运价管理员确认。**

### 业务价值：**HIGH**

| 维度 | 当前 | 目标 |
|------|------|------|
| 单组合成本核算 | 10 分钟（手工 Excel） | 1 分钟（系统自动计算） |
| 全量运价更新 | 1 天（逐一修改） | 0.5 天（批量更新） |
| 成本可追溯性 | 无（Excel 覆盖即丢失） | 100%（操作审计日志完整留痕） |
| 报价一致性 | 低（销售各自 Excel） | 高（统一公布价 + 优惠规则） |

## 2. 用户故事 (User Story)

### 服务渠道 (ServiceChannel)

服务渠道是超级运价模块的**最基础实体**。所有服务组合、成本、定价、优惠均挂在服务渠道之下。渠道配置是定价引擎运转的前提。

> **As a** 运价管理员
> **I want to** 创建一个服务渠道，定义其运输方式、尾程派送、包税方式等基本属性
> **So that** 后续的服务组合生成、成本核算、运价查询均有统一的渠道框架约束

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道名称 | 文本 | ✅ | 如"美森360" |
| 起运国 | 单选 | ✅ | 下拉选项调用基础资料-国家接口；如"中国" |
| 目的国 | 多选 | ✅ | 下拉选项调用基础资料-国家接口；如"美国""英国"，**控制 FBA 仓库下拉范围** |
| 运输方式 | 单选 | ✅ | 海运/空运/洲际火车/洲际卡车 |
| 尾程派送方式 | 多选 | ✅ | 卡派/快递派 |
| 包税方式 | 多选 | ✅ | 包税/不包税/自主税号/自税递延 |
| 计费方式 | 多选 | ✅ | 重量KG/体积m³ |
| 服务类型 | 多选 | ✅ | 散货/整柜 |
| 是否需要时效理赔 | 单选 | ✅ | Y/N |

> **约束**：尾程派送方式、计费方式创建后，该渠道下服务组合的对应选项受此约束。

#### B. 业务规则：目的国 × FBA 仓库过滤

FBA 仓库主数据需包含 `country` 字段（如 ONT8→美国，BHX4→英国）。服务渠道的目的国选择控制下游时效配置中的 FBA 仓库可选范围：

- 配置送仓预估时效/理赔承诺时效时，FBA 仓库下拉仅展示 `仓库.country ∈ 服务渠道.目的国` 的仓库
- 目的国减少时，如已有该国 FBA 仓库的时效配置，阻止操作并提示"需先清理该国仓库的时效配置后，方可移除此目的国"

---

### 预估运输时效 (EstimatedTransitTime)

挂在服务渠道上，**1:1** 关系。向亚马逊推送预计送达时段（Delivery Window），直接影响 FIST 考核的 DW 准确率。

> **As a** 运价管理员
> **I want to** 为服务渠道配置是否向亚马逊推送 FBA 运单 DW 及对应的服务等级和运输天数
> **So that** 亚马逊 FIST DW 考核有据可依，推送数据准确

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道 | channel_id | ✅ | |
| FBA运单同步 | Boolean | ✅ | 默认=否；控制是否向亚马逊推送 DW |
| 亚马逊服务等级 | Enum | 条件必填 | `标准` / `加急` / `快速`；FBA运单同步=是时必填 |
| 运输最小天数 | Integer | 条件必填 | 单位：天；FBA运单同步=是时必填 |
| 运输最大天数 | Integer | 条件必填 | 单位：天；FBA运单同步=是时必填，须 ≥ 运输最小天数 |

#### B. 业务规则

FBA运单同步=否时，另三个字段不存储（而非存储后 disabled）。如果未来切换为"是"，运价管理员必须主动重新填写——避免"过期数据静默激活"的风险。

---

### 送仓预估时效 (WarehouseDeliveryEstimate)

挂在服务渠道上，**1:N**（每个 FBA 仓库一条记录）。分两阶段向亚马逊更新送仓预估，提高 DW 准确率。

> **As a** 运价管理员
> **I want to** 为每个 FBA 仓库配置首次预估时效和提柜后的二次修正时效
> **So that** DW 分两阶段滚动更新，提柜后修正为更精确的送达时间

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道 | channel_id | ✅ | |
| FBA仓库代码 | String | ✅ | 受目的国过滤，下拉仅展示 `仓库.country ∈ 服务渠道.目的国` 的仓库 |
| 预估送仓时效（天） | Integer | ✅ | 第一阶段：运单创建时，DW = 开船日 + 此值，首次推送给亚马逊 |
| 提柜至送仓（天） | Integer | ✅ | 第二阶段：提柜后，DW = 提柜日 + 此值，重新预估并更新至亚马逊 |

#### B. 业务规则：DW 滚动更新

```
┌─────────────────────────────────────────────────┐
│  第一阶段（运单创建时）：                          │
│    DW = 开船日 + 预估送仓时效（天）               │
│    → 首次推送给亚马逊                              │
│                                                   │
│  第二阶段（运单提柜后）：                          │
│    DW = 提柜日 + 提柜至送仓（天）                  │
│    → 重新预估并更新至亚马逊                        │
└─────────────────────────────────────────────────┘
```

> **业务价值**：首次预估偏保守（用"预估送仓时效"），提柜后有了精确的提柜时间，用"提柜至送仓"做二次修正。两阶段滚动更新能显著提升 DW 准确率——这是 FIST Elite 承运商的核心考核指标。

#### C. 校验规则

提柜至送仓 ≤ 预估送仓时效，否则阻止保存并提示"提柜至送仓天数不可超过预估送仓时效"。

---

### 理赔承诺时效 (ClaimCommitment)

挂在服务渠道上，**1:N**（按派送方式 + FBA 仓库分组）。后续运单模块根据实际送达天数 vs 理赔承诺时效，自动判定是否需要理赔。

> **As a** 运价管理员
> **I want to** 按派送方式和 FBA 仓库配置理赔承诺时效
> **So that** 运单延误理赔有标准可查，系统自动判定触发条件

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道 | channel_id | ✅ | |
| 派送方式 | Enum | ✅ | `快递派` / `卡派`；从服务渠道的尾程派送方式中取值 |
| FBA仓库代码 | 多选 | 条件必填 | 卡派必填（不同仓点可配置不同理赔时效）；快递派非必填（快递网络覆盖全美，时效相对一致） |
| 理赔时效（工作日） | Integer | ✅ | 单位：工作日；实际送达天数 > 此值 → 触发理赔判定 |

#### B. 业务规则

- 服务渠道 `是否需要时效理赔` = N → 理赔承诺时效配置不可编辑，已有数据清空
- 卡派未选仓库 → 阻止保存并提示"卡派必须指定 FBA 仓库"
- 同一仓点不可重复出现在多条理赔规则中

### 港口 (Port)

港口数据属于基础数据模块，不在超级运价内维护。超级运价通过 `port_code` 引用。航线中的出发港/目的港字段存储的是港口代码，非中文名。

#### A. 字段定义

| 字段 | 类型 | 说明 |
|------|------|------|
| 港口名称 | String | 如"上海港"、"长滩港" |
| 港口代码 | String | 唯一标识，如 `USLAX`、`CNSGH` |
| 国家 | String | 下拉选项调用基础资料-国家接口；如"中国"、"美国" |
| 类型 | Enum | 出发港 / 目的港 |

---

### 航线 (Route)

航线是连接"报价端"和"亚马逊推送端"的关键桥梁——提单绑定航线，运单绑定提单，运单推送到亚马逊 FIST 时从提单→航线获取港口代码。

> **As a** 系统
> **I want to** 运单推送亚马逊 FIST 时，从提单关联的航线中自动获取出发港代码和目的港代码
> **So that** 关键事件节点（干线出发/到达）的港口信息准确且格式符合亚马逊要求

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 航线代码 | String | | 船司分配的航线代码，如 `AAC/AAC2`、`CLX`、`SEA/SEA3`。默认为空，可编辑。同一船司的同一港口对可能有多条航线代码（如 COSCO 宁波→长滩同时有 AAC/AAC2 和 AAC/AAC4/PS5），因此不能用出发港+目的港拼接 |
| 出发港 | port_code | ✅ | 引用港口主数据，展示格式 `港口名称（港口代码）`如"长滩港（USLAX）"，数据库存名称+代码 |
| 目的港 | port_code | ✅ | 同上 |
| 船司 | supplier_id | ✅ | 引用 SRM 供应商管理模块（供应商大类"干线运输供应商"），不在超级运价内重复维护 |
| 预计运输天数 | Integer | | 配置在航线级别，船期可覆盖 |

#### B. 关联链路

航线 N:N 关联服务渠道（用于报价展示），1:N 关联提单（运单推送 FIST 时自动获取港口代码）。

```
港口主数据 ──→ 航线 ──→ 服务渠道（N:N，用于报价展示）
              │
              └──→ 提单（1:N，提单绑定一条航线）
                     │
                     └──→ 运单（1:N，运单绑定一张提单）
                            │
                            └──→ Amazon FIST 推送
                                   ├── 干线出发事件 → 取 航线.出发港代码
                                   └── 干线到达事件 → 取 航线.目的港代码
```

> 一条渠道可配多条航线（如快船600同时有 SEA/SEA3 和 AAC/AAC2 等航线）。一个提单只绑定一条航线。Amazon FIST 推送时，运单通过提单→航线直接拿到港口代码，无需在运单层面重复维护港口信息。

### 周船期批次 (WeeklyScheduleBatch)

运价管理员的工作粒度是"周"——每周一早上同步更新所有航线的本周船期和运价。周船期批次是按周汇总的主列表，作为「运价更新工作台」的核心入口。

> **As a** 运价管理员
> **I want to** 按周创建船期批次，系统自动管理批次状态
> **So that** 每周的船期和运价属于同一运维周期，查询时只展示当前周数据

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 批次名称 | String | ✅ | 如 "2026-W05（1/26-2/1）" |
| 周次标识 | String | ✅ | ISO 周次，如 `2026-W05`，唯一索引 |
| 周起始日 | Date | ✅ | 周一日期 |
| 周结束日 | Date | ✅ | 周日日期 |
| 状态 | Enum | ✅ | PREVIEW（预告）/ CURRENT（当前）/ EXPIRED（已过期） |
| 备注 | String | | |

#### B. 业务规则：状态流转

每周日 24:00 系统自动将 CURRENT → EXPIRED，下周的 PREVIEW → CURRENT。同一时刻只有一条 CURRENT 批次。运价查询时只展示 CURRENT 周的船期。

```
│  ...                   ...      ...      ...    ...     ...   ...        │
│                                                                           │
│  [新增航线船期]  [从上周复制]  [批量导入]                                   │
└───────────────────────────────────────────────────────────────────────────┘
```

### 船期 (SailingSchedule)

每条船期记录挂在周批次下，关联一条航线。船期表是运价查询时销售必看的信息——"这周有没有船？什么时候开？什么时候到？"

> **As a** 销售
> **I want to** 在运价查询结果中看到每个渠道的船期信息（船司、航线、ETD、ETA）
> **So that** 客户问"我的货什么时候能到"时，当场给出准确答复

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 周批次 | batch_id | ✅ | 所属周批次 |
| 航线 | route_id | ✅ | 关联航线 |
| 截单时间 | DateTime | ✅ | 按本地仓库配置，每个仓库一个截单时间（如广州仓、汕头仓、义乌仓各自独立），仓库引用仓库列表的"仓库名称" |
| 预计开船时间 (ETD) | Date | ✅ | |
| 预计到港时间 (ETA) | Date | ✅ | |
| 航程（自然日） | Integer | ✅ | ETD→ETA 天数 |
| 最快提取/送仓时间 | Date | | 开船后第 X 天 |

#### B. 业务规则：双视角展示

周船期主列表是"运维视角"（本周有哪些船），航线历史是"追溯视角"（这条航线过去/未来的船期变化）。运价更新工作台默认展示本周主列表，点击航线名称可展开该航线的历史船期。

### 重量段配置 (WeightTier)

挂在服务渠道上，该渠道下所有组合共享。每个服务渠道按计价单位（KG / CBM）分别定义自己的分段阶梯，不同渠道阶梯不同。

> **As a** 运价管理员
> **I want to** 为每个服务渠道按 KG 和 CBM 分别定义计费分段
> **So that** 不同渠道和不同计价方式可使用不同的分段规则

#### A. 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道 | channel_id | ✅ | 挂在哪个服务渠道下 |
| 计价单位 | Enum | ✅ | `KG` / `CBM`；与服务渠道.计费方式联动——计费方式含 KG 则必须配置 KG 段，含 CBM 则必须配置 CBM 段 |
| 段名称 | String | ✅ | 如"8KG+"、"101KG+"、"2CBM+" |
| 起始值 | Decimal | ✅ | 该段起始值（含）；KG 单位为 KG，CBM 单位为 m³ |
| 结束值 | Decimal | | 该段结束值（不含）；null = 以上 |

示例：
```
美森360：
  KG 段:  8KG+（8~101KG）、101KG+（101KG 以上）
  CBM 段: 2CBM+（2m³ 以上）

美国空运240：
  KG 段:  12KG+（12~21KG）、21KG+（21~51KG）、51KG+（51~101KG）、101KG+（101KG 以上）

国际快递（UPS/DHL）：
  KG 段:  23kg+（23~47kg）、47-72kg、73-99kg
```

### 运价行 (PriceTableRow)

公布价不是单一数字，而是**仓库组 × 重量段 × 交货地**的三维矩阵。每条运价行对应一个"价格 SKU"。

> **As a** 运价管理员
> **I want to** 为每个服务组合按仓库组 × 重量段 × 交货地维护公布价
> **So that** 系统能根据销售输入的货物参数精确匹配对应价格

#### A. 字段定义

运价行挂在周船期批次下，与船期同频每周维护。运价查询时只取 CURRENT 批次的运价行。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 周批次 | batch_id | ✅ | 所属周船期批次；与船期同频每周更新 |
| 服务组合 | combination_id | ✅ | |
| 仓库组 | String (逗号分隔) | ✅ | 如 `ONT8,LAX9,LGB8` |
| 重量段 | weight_tier_id | ✅ | 渠道级定义的重量段 |
| 交货地 | pickup_region | ✅ | 珠三角/汕头/义乌 |
| 公布价-per_KG | Decimal | 条件必填 | 若计费方式含 KG |
| 公布价-per_CBM | Decimal | 条件必填 | 若计费方式含 CBM |

> **运价周期**：新建周批次时从上周复制全部运价行，运价管理员调价后发布。查询时只取 CURRENT 批次。
>
> **不是存储字段**：`参考签收时效` 和 `理赔时效` 不存储在运价行上，查询时自动计算：
> - **参考签收时效** = 当前周船期.最快提取/送仓时间 换算为自然日范围（或 = 船期.航程 + 送仓预估时效.预估送仓时效）
> - **理赔时效** = 通过 运价行.服务组合 → 服务渠道 → 理赔承诺时效，按派送方式 + FBA仓库匹配取值

#### B. 与成本底价的关系

```
成本底价(组合, 仓点A) = 头程共享 + 派送段(仓点A)
成本底价(组合, 仓点B) = 头程共享 + 派送段(仓点B)

运价行(组合, 仓库组="A,B,C", 重量段, 交货地) → 公布价
  └── 仓库组覆盖仓点 A、B、C，取三个仓点中成本最高的作为定价基准
  └── 利润 = 公布价 - max(成本底价A, 成本底价B, 成本底价C)
```

> **设计决策**：仓库组用逗号存而非独立表——当前分组稳定，15 个渠道约需 50-80 个仓库组，逗号方案简单够用。未来如果需要仓库组独立管理（如改名、移动仓点），再迁移至独立表。

### Epic 1：成本底价管理

> **As a** 运价管理员
> **I want to** 自定义成本段和成本项，服务渠道选择关联的段+项并设默认单价，服务组合从渠道继承后可独立禁用不适用的项或覆盖单价
> **So that** 成本结构可灵活适配不同业务模式，组合维度的成本项可精细化管理，且成本底价变动有据可查

**成本模型（四层结构）**：

```
成本段（全局目录）  →  成本项模板（全局目录，挂段）  →  渠道成本项（渠道选择+定价）  →  组合成本项（继承+覆盖）
─────────────────     ─────────────────────────     ─────────────────────────     ─────────────────────────
Level 1: 定义段名       Level 2: 定义项名+公式        Level 3: 选哪些段/项+默认价    Level 4: 继承后可调价/禁用
全局复用，可CRUD        全局复用，可CRUD              每个渠道独立配置              每个组合独立维护
```

```
成本底价(组合, 仓点) = 头程共享成本 + 派送段成本(按仓点)

头程共享成本（非派送段，所有仓点共用）：
  揽收段 + 干线运输段 + 关务段 + 仓储段 + ...
  → 按 KG标准 / CBM标准 均摊为 per_KG 和 per_CBM 单价

派送段成本（按仓点独立）：
  每个 FBA 仓点有三种供应商报价来源：
    ├── 拆送价（按每箱派送）
    ├── 组合一口价（按整板派送）
    └── 整柜直送价（仅整柜适用）
  → 取 MAX(拆送价, 组合一口价) 作为散货卡派的派送段成本底价
  → 拆送 vs 组合仅为内部计算过程，对外不暴露

最终成本底价 = 头程共享 per_KG + 派送段 per_KG（按仓点）
              └── 粒度：服务组合 × FBA仓点
```

| 字段 | 类型 | 必填 | 说明 |

| 排序 | Integer | ✅ | 控制展示顺序 |

#### 渠道成本项 (ChannelCostItem) — 渠道级默认值

服务渠道从全局目录中选择要使用的段和项时，系统自动创建渠道成本项记录。选中项自动带出模板的 `建议默认价`，渠道可覆盖。**渠道成本项是纯数据层实体，无独立管理页面**——运价管理员只需在服务渠道编辑页完成段+项的选择和调价。服务组合创建时从此复制。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务渠道 | channel_id | ✅ | 系统自动关联 |
| 成本段 | segment_id | ✅ | 继承自渠道选择 |
| 成本项模板 | item_template_id | ✅ | 继承自渠道选择 |
| 原始单价 | Decimal | ✅ | 继承模板建议默认价，渠道可覆盖（如美森海运费 5.2 vs 快船 3.0） |
| 币种 | Enum | ✅ | 继承模板币种，可覆盖 |
| 交货地 | pickup_region | 条件必填 | 成本段=揽收段时必填 |
| 派送报价来源 | Enum | 条件必填 | 成本段=派送段时必填：`拆送价` / `组合一口价` / `整柜直送价` |

> 渠道成本项是"默认值"，不是"共享引用"。修改此处不会自动影响已有组合——需通过批量更新流程主动推送。

#### 组合成本项 (CombinationCostItem) — 组合级实际值

服务组合创建时从渠道成本项**逐一复制**。每个组合可独立：覆盖单价、禁用不适用的项。成本底价计算时仅统计启用的项。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务组合 | combination_id | ✅ | |
| 来源渠道成本项 | channel_cost_item_id | | 可追溯到渠道级的哪个默认项 |
| 成本段 | segment_id | ✅ | |
| 成本项名称 | String | ✅ | 复制自模板 |
| 原始单价 | Decimal | ✅ | 继承渠道默认值，可单独覆盖 |
| 币种 | Enum | ✅ | `RMB` / `USD` |
| 汇率 | Decimal | 条件必填 | 币种=USD 时必填，用于美元换汇计算 |
| cost_per_kg | Decimal | | 系统自动计算 |
| cost_per_cbm | Decimal | | 系统自动计算 |
| 交货地 | pickup_region | 条件必填 | 成本段=揽收段时必填 |
| 派送报价来源 | Enum | 条件必填 | 成本段=派送段时必填 |
| FBA仓库代码 | String | 条件必填 | 成本段=派送段时必填 |
| 是否启用 | Boolean | ✅ | 默认启用；禁用后该组合不适用此成本项，计算时跳过 |
| effective_from | DateTime | | 未来生效时间；一期预留字段，手动切换 |

> **计算公式**：详见 [4. 核心算法 — STEP 1](#4-核心算法最终报价计算引擎)。派送段取 MAX(拆送价, 组合一口价) 作为成本底价，整柜取直送价。

1. 初始化：系统预置 5 段 + 常用项模板（海运费、报关费、拆送价、组合一口价等），运价管理员可按需新增/禁用

### Epic 3：利润策略与公布价

> **As a** 运营主管
> **I want to** 为服务组合配置利润策略（固定金额加成 / 百分比加成）
> **So that** 公布价自动基于成本底价生成，无需人工逐项加利润

#### 利润策略字段

利润策略挂在服务组合上，**1:1** 关系：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 服务组合 | combination_id | ✅ | |
| 加成方式 | Enum | ✅ | `固定金额加成` / `百分比加成` |
| 加成值 | Decimal | ✅ | 固定金额：元/KG 或 元/CBM；百分比：如 15 表示 15% |
| 加成币种 | Enum | 条件必填 | 加成方式=固定金额加成时必填：`RMB` / `USD` |
| scope_type | Integer | ✅ | 一期固定 = 10（组合级）；预留 20（段级）/ 30（项级） |

> **计算公式**：公布价 = 成本底价 × (1 + 加成百分比%) 或 成本底价 + 固定金额。KG 和 CBM 分别计算。scope_type 一期仅处理 10，二期扩展段级/项级。

**核心场景**：
1. 配置利润策略（组合级，scope_type=10），选择固定金额加成（如 +2元/KG）或百分比加成（如 +15%）
2. 系统根据利润策略 + 成本底价自动计算公布价，写入对应运价行
3. 新建周批次时，从上周复制运价行并基于最新成本底价重新计算公布价，运价管理员可覆盖后发布
4. 运价查询时公布价按 KG 和 CBM 双计价展示

### Epic 4：附加费规则引擎（配置端）

> **As a** 运价管理员
> **I want to** 灵活定义附加费规则（触发条件 + 计价公式 + 费用方向），系统在运单创建时自动匹配并计算
> **So that** 不再用 Excel 逐一判断哪些运单需要加收附加费，消除人工遗漏和计算错误

#### 规则类型概述

附加费规则按触发机制分为三种类型，不同类型对应不同的配置表单和匹配逻辑：

| 类型 | 触发机制 | 典型场景 |
|------|---------|---------|
| **字段匹配** | 运单/货物字段 = 某值 | 商品属性=粉末、地址类型=商业、邮编∈偏远列表、自送=是 |
| **公式阈值** | 计算公式结果 > / < 阈值 | (长+宽)×2+高>300 或 长+宽×2+高×2>265 |
| **人工添加** | 运单界面人工勾选 + 填数量 | 贴标费、报关费、周六派送加收 |

---

#### 附加费规则 (SurchargeRule) — 通用字段

三种类型共享的字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 规则名称 | String | ✅ | 唯一标识，如"粉末品名加收" |
| 规则类型 | Enum | ✅ | `字段匹配` / `公式阈值` / `人工添加` |
| 动作类型 | Enum | ✅ | `加收费用`（收费）/ `标识提醒`（运单打标不收费）/ `阻止下单`（拦截） |
| 触发节点 | Enum | ✅ | `运单创建` / `红外过机`；公式阈值型典型配为红外过机 |
| 费用类型标签 | String | ✅ | 用于报表分类，如"出口费用""国内费用" |

### Epic 5：运费优惠与特批价管理（配置端）

#### 业务规则

**优惠叠加规则**：

- 优惠值 > 0 表示加价（如特批加价 10% = 1.10 或 +2元/KG）
- 同一客户命中多条规则 → 折扣率模式乘法叠加，固定减免模式加法叠加；先算折扣率再算固定减免
- 适用范围匹配优先级：客户特批 > 客户等级 > 全局；仓点指定 > 渠道指定 > 全局

**生命周期管理**：

```
PENDING（飞书审批中）
  → 审批通过 → 运价管理员录入系统 → ACTIVE
  → 审批拒绝 → 不录入

ACTIVE
  → 到期日到达 → 定时任务自动置为已过期
  → 到期前 7 天 → 系统推送提醒给运价管理员
  → 单次有效类型首次被运单引用 → 自动已过期

已过期（终态）
  → 运价查询不再展示
  → 历史运单仍可追溯原优惠
```

**价格底线管控（BPR-1）**：

当运价管理员录入特批价、或销售查询运价时，系统计算最终报价后自动显示盈亏标识：

| 标识 | 条件 | 含义 | 行为 |
|------|------|------|------|
| 绿灯 | 最终报价 ≥ 成本底价 × 1.05 | 正常盈利 | 正常 |
| 黄灯 | 成本底价 ≤ 最终报价 < 成本底价 × 1.05 | 薄利预警 | 销售可见，提示"薄利" |
| 红灯 | 最终报价 < 成本底价 | 亏本 | 运价查询界面标红，运价管理员录入时弹窗确认 |

> **设计决策**：红灯不强制拦截（因为有飞书审批兜底），但在系统层面做"知会型管控"——让所有人知道这单在亏钱。如果未来需要强制拦截，仅需在查询接口层加一个开关。

### Epic 6：附加费优惠管理（配置端）

> **As a** 运价管理员
> **I want to** 为客户配置附加费优惠（与运费优惠独立），系统在匹配附加费后自动应用
> **So that** 客户在享受运费折扣的同时，附加费也能享受相应优惠，运费优惠和附加费优惠独立计算不互相干扰

#### 附加费优惠规则 (SurchargeDiscount) — 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 规则名称 | String | ✅ | 唯一标识，如"VIP-粉末附加费减0.5元" |
| 优惠对象类型 | Enum | ✅ | `客户等级` / `客户特批` |
| 目标客户等级 | String | 条件必填 | 优惠对象=客户等级时必填 |
| 目标客户 | customer_id | 条件必填 | 优惠对象=客户特批时必填 |
| 目标附加费规则 | surcharge_rule_id | ✅ | 绑定到哪条附加费规则 |
| 优惠方式 | Enum | ✅ | `折扣率` / `固定减免`（如粉末加收 1.5 → 减 0.5 = 实收 1.0） |
| 优惠值 | Decimal | ✅ | 折扣率模式：0.01~9.99；固定减免模式：元，如 -0.5 |
| 有效期类型 | Enum | ✅ | `永久有效` / `固定期限` / `单次有效` |
| 有效期起 | Date | 条件必填 | 类型=固定期限时必填 |
| 有效期止 | Date | 条件必填 | 类型=固定期限时必填 |
| 状态 | Enum | ✅ | `生效中` / `已过期` |

---

#### 业务规则

**与运费优惠的关系**：

系统在创建附加费优惠规则时，校验目标附加费规则的 `discountable_level`：

| discountable_level | 含义 | 优惠限制 |
|-------------------|------|---------|
| FULL | 服务型附加费（如贴标费） | 可全免，折扣率 0~∞ |
| PARTIAL | 成本型附加费（如粉末加收） | 最低收成本价，折扣率 ≥ 成本占比 |
| NONE | 代收代缴型（如政府税费） | 不可配置任何优惠 |

> **一期实现**：建表时预留 `discountable_level` 字段，创建优惠规则时做软校验（弹窗提示，不强制拦截）。二期完善硬拦截逻辑。

```
最终报价 = (公布价 × Σ运费折扣率 + Σ运费固定减免) + Σ(附加费金额 × Σ附加费折扣率 + Σ附加费固定减免)

两类优惠独立计算，互不影响：
  - 折扣率模式：乘法叠加（折上折）；固定减免模式：加法叠加
  - 先算折扣率，再算固定减免
  - 运费优惠不影响附加费，附加费优惠不影响运费
```

**可优惠性校验（BPR-4）**：

**案例还原（来自实际业务）**：

> 客户：VIP 等级，产品粉末
> 美森360 ONT8 卡派，公布价 9元/KG
>
> 运费部分：
>   - VIP 等级折扣 9折 → 9 × 0.9 = 8.1元/KG
>
> 附加费部分：
>   - 粉末加收：+1.5元/KG
>   - VIP 附加费优惠：粉末附加费 -0.5元/KG（折扣率 0.67）
>   - 实际粉末附加费：1.5 × 0.67 = 1.0元/KG
>
> 最终报价 = 8.1 + 1.0 = 9.1元/KG

### Epic 7：运价查询与最终报价（销售视图 — 反向匹配引擎）

> **As a** 销售
> **I want to** 输入货物属性（产品类型、件数、重量、尺寸、目的国、目的仓），系统自动匹配所有符合条件且可承运的服务渠道及报价
> **So that** 客户说"3 件粉末，每件 60KG，发 ONT8"时，我当场就能对比美森360 vs 美森432 vs EX456 的多维度报价

#### 7A. 查询入口

```
销售输入（必须）：
  ├── 目的国          → 过滤服务渠道.目的国
  ├── 目的仓点        → 过滤送仓预估时效中有该仓点的渠道
  ├── 件数            → 过滤承运范围（最小件数 ≤ 件数 ≤ 最大件数）
  ├── 单件重量/尺寸   → 计算总重量、总体积、材重
  ├── 计费方式偏好    → KG / CBM（可多选）
  └── 产品类型（可选）→ 匹配附加费规则 + 产品限制

销售输入（可选）：
  ├── 交货地          → 匹配运价行的 pickup_region，默认珠三角
  └── 客户名称        → 匹配运费优惠 + 附加费优惠
```

#### 7B. 匹配流程

```
Step 1 — 渠道初筛
  Filter: 服务渠道.目的国 contains 输入的目的国
  Filter: 送仓预估时效 contains 输入的目的仓点
  Filter: 件数 ∈ [最小件数, 最大件数]
  → 候选渠道列表

Step 2 — 组合匹配
  对每个候选渠道，找到其所有 ACTIVE 服务组合
  Filter: 计费方式 match 输入偏好
  → 候选组合列表

Step 3 — 运价行匹配
  对每个候选组合：
    Filter: 运价行.周批次 = CURRENT（仅查本周运价）
    Filter: 仓库组 contains 输入的目的仓点
    Filter: 重量段 match（重量段.计价单位=KG 时匹配计费重量，=CBM 时匹配计费体积）
    Filter: 交货地 = 输入的交货地
    → 取匹配的运价行.公布价
    → 如未匹配（重量不在任何段内），标记"需人工报价"

Step 4 — 附加费匹配
  对命中的组合，匹配附加费规则：
    Filter: 产品类型触发附加费（如"粉末"→ +1.5元/KG）
    Filter: 货物属性触发（超重超长等）
    → 计算附加费明细

Step 5 — 优惠叠加（若选择了客户）
  匹配运费优惠 → 折上折乘法叠加
  匹配附加费优惠 → 独立计算

Step 6 — 船期关联
  对每个命中渠道，查询 CURRENT 周船期批次中的船期（ETD、ETA、航程）
  → 展示"下一班船：X 月 X 日出发，预计 X 月 X 日到港"

Step 7 — 底线检查
  成本底价 = 头程共享 + 派送段(该仓点)
  盈亏标识 = GREEN / YELLOW / RED
```

### NFR（非功能性需求）

- **性能**：运价查询（含优惠计算）响应时间 < 3 秒；批量更新 14 个组合 < 1 分钟
- **并发**：一期支持 5-10 个运营人员同时操作，无需考虑高并发
- **数据保留**：审计日志保留 ≥ 2 年；运价历史版本保留 ≥ 1 年
- **精度**：金额精度 0.01 元（CNY/USD）；重量精度 0.1KG；尺寸精度 1CM
- **安全**：成本底价仅运价管理员可编辑；公布价销售只读；审计日志不可删除

## 4. 计算示例：最终报价完整链路

```
输入: {产品=粉末, 目的国=美国, 目的仓=ONT8, 件数=3, 单件60KG, 60×60×60, 交货地=珠三角, 客户=VIP}

Step 1 — 渠道初筛: 目的国=美国 → 命中美森360、美森432、EX456等
          仓点=ONT8 → 命中送仓预估时效含ONT8的渠道
          件数=3 → 在承运范围内
          → 候选渠道: [美森360, 美森432, EX456, ...]

Step 2 — 成本底价:
          头程共享 = 8.6/KG
          派送段ONT8拆送=0.60, 组合=1.11 → MAX = 1.11
          cost_per_kg = 8.6 + 1.11 = 9.71

Step 3 — 公布价（系统计算 → 写入运价行 → 查询时匹配）:
          利润策略: 美森360 组合级固定加成 +2元/KG
          系统生成: 9.71 + 2 = 11.71，运价管理员调整为 10.0 写入运价行
          匹配: 仓库组"ONT8,LAX9,LGB8" contains ONT8 ✅
                重量段: 总重量=180KG → 匹配 101KG+ 段
                交货地=珠三角
          → published_per_kg = 10.0 元/KG

Step 4 — 运费优惠:
          VIP折扣率 0.9 → 10.0 × 0.9 = 9.0
          或 VIP固定减免 -1元/KG → 10.0 - 1.0 = 9.0
          discounted_per_kg = 9.0

Step 5 — 底线检查: 10.0 < 9.71 × 1.05 = 10.20 → YELLOW（薄利）
          ⚠️ 公布价 10.0 接近成本 9.71，边际仅 3%

Step 6 — 附加费匹配: 产品类型=粉末 → 命中"粉末品名加收" +1.5/KG
          报关方式未指定 → 暂不匹配报关费

Step 7 — 附加费优惠: VIP附加费固定减免 -0.5元/KG → 1.5 - 0.5 = 1.0/KG

Step 8 — 最终报价:
          运费优惠后 9.0 + 粉末加收优惠后 1.0 = 10.0 元/KG 🟡
          
多渠道对比:
  公布价  运费优惠后  附加费优惠后  最终报价  船期
  美森360: 10.0 → 9.0 + 1.0 = 10.0/KG 🟡 | 1/28→2/8(11天)
  美森432: 9.7  → 8.7 + 1.0 = 9.7/KG  🟢 | 1/29→2/9(11天)
  EX456:   5.6  → 5.0  + 1.0 = 6.0/KG  🟢 | 1/28→2/11(14天) ← 最便宜但最慢
```

## 7. 专家建议

### 建议 1：把"运价更新工作台"做成 MVP 的门面功能

这是每周使用频率最高的入口。建议作为**首页 Dashboard** 或**顶部独立 Tab**。核心路径：

```
登录 → 运价更新工作台（首页）
     → 本周船期主列表（WeekScheduleBatch，展示所有航线本周船期一览）
     → 看到"本周待更新成本项"列表（海运费标红）
     → 批量修改 → 展示受影响组合 → 勾选 → 一键重算
     → 同一页面中可切换"船期"和"运价"两个 Tab
```

**设计原则**：高频操作 ≤ 2 步，低频配置可接受 3-4 步。船期和运价在同一工作台维护，因为二者是同频的——船公司周一发来新海运费时，往往也同步更新船期。

### 建议 2：成本批量更新用"成本项模板 → 勾选推送到关联组合"模式

修改成本项模板的建议默认价后，展示"受影响的服务组合清单"（列出当前公布价），让管理员**勾选后批量刷新**。不做无脑全量刷新——可能存在某些组合正在谈价，暂不想变动。

### 建议 3：特批价流程不做系统内审批

飞书审批链已成熟运行。系统只保留"录入特批价结果"功能，关联飞书审批编号即可。(1) 一期开发量可控；(2) 0-1 阶段不碰 BPM 工作流。

### 建议 4：EXCEL 派送段数据做"结构化清洗"

《美线头程成本核算表》中，一个仓点行包含 6 组数据（拆送/组合/直送 × KG/CBM）及末端/头程拆分。导入时引导运价管理员按仓点逐行录入，系统自动算 MAX，不直接套 Excel 格式。

### 建议 5：预留附加费应收-应付关联字段

附加费规则（Epic 4）一期预留 `cost_item_id` 字段（关联供应商应付成本项），二期 BPR-2 填数据做毛利率分析。一期存 NULL，SQL 成本极低；二期改表成本极高。
