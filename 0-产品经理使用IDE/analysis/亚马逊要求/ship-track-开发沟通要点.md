# Ship Track — 开发沟通要点

> 基于 Amazon FIST-ShipTrack API 文档 + ship truck 项目需求 + RDD §3.2-§3.3
> 日期：2026-06-07 | 状态：待沟通

---

## 一、背景

飞点作为亚马逊 FIST 承运商，需通过 ShipTrack API 管理运单的预计送达时段（Delivery Window, DW）。亚马逊对 DW 修改有严格的时间窗口校验——**不允许先拿预约后改 DW**。如果预约日距离当前 DW 窗口太远，API 会拒绝修改请求（BLOCK）。

## 二、当前设计（RDD §3.2-§3.3）

当前定义了 DW 的两阶段滚动更新：

| 阶段 | 触发节点 | DW 计算 | 说明 |
|------|---------|--------|------|
| 第一阶段 | 运单创建时 | DW = 开船日 + 开船至送仓（天） | 首次推送给亚马逊 |
| 第二阶段 | 运单提柜后 | DW = 提柜日 + 提柜至送仓（天） | 重新预估并更新至亚马逊 |

**缺失：第三阶段 — 预约确认后的最终 DW 同步。**

## 三、ship truck 需求文档要求的完整 SOP

| 阶段 | 关键动作 | 系统动作 | DW 精度 |
|------|---------|---------|---------|
| 1. 接单与预报 | 接收订单，填入初始时间窗 | 创建货件时填入初始 DW | 低（客户预估） |
| 2. 启运后更新 | 货物开船/起飞 | 系统自动：开船日 + 渠道 + 仓点 + 平均海运时效 | 中 |
| 3. 到港/清关后更新 | 清关完成 | 系统自动：清关完成日 + 平均清关时效 | 高（接近真实） |
| 4. **预约确认** | 卡车公司提供预约确认单，运营 2 小时内录入 | **系统调用 API 将 DW 修改为以预约日为中心的窗口** | **精确** |
| 5. 派送执行 | 监控 GPS，获取 POD | 系统标记已送达 | — |

## 四、需要与开发对齐的 4 个问题

### 问题 1：DW 更新时序与 API 约束

亚马逊 API 对 DW 修改有时间窗口限制。如果先拿预约、后调 API 改 DW，大概率被 BLOCK。

**需确认：**
- 亚马逊 DW 修改的允许时间窗口是什么？（预约日前 N 天？开船后 M 天？）
- 我们的两阶段滚动更新（开船→提柜）能否保证预约日始终在允许范围内？
- 是否需要调整为"三阶段"（开船→清关→预约确认）？

### 问题 2：被 BLOCK 时的兜底流程

需求文档要求"触发人工处理流程"。当前系统设计没有这个分支。

**需确认：**
- BLOCK 后谁来处理？（运价管理员？运营？客服？）
- 通知方式？（系统内弹窗？飞书消息？）
- 人工处理的具体操作是什么？（手动调 DW？联系亚马逊？重新拿约？）
- 是否需要"被 BLOCK 运单池"的页面来集中管理？

### 问题 3：飞书表格 → TMS 的数据流

当前流程依赖"卡车公司 → 飞书表格 → 运营手动录入系统"。这是过渡方案还是长期方案？

**需确认：**
- 是否需要系统直接对接卡车公司的预约确认接口？（跳过飞书表格）
- 如果飞书表格是长期方案，是否需要系统定时轮询飞书表格？
- 运营"2 小时内录入"的 SLO 如何保障和监控？

### 问题 4：RDD / 数据设计 / PRD 的补充

**需确认后更新文档：**
- RDD §3.2-§3.3 补充第三阶段"预约确认→最终 DW"
- 数据设计补充：预约确认时间字段、DW 被 BLOCK 状态、人工处理记录
- PRD §5.1 运价更新工作台可能需要新增"被 BLOCK 运单提醒"区域

## 五、沟通结论（2026-06-07）

### 结论 1：DW 时间窗获取方式

**新流程**：海外仓 → 给运营 DW 时间窗 → 运营登记在飞书表格 → 系统每天定时拉取 → 调用 Amazon API 修改 DW → 推送。

- **获取 DW 条件**：提单号 + 仓点代码
- **错误处理**：API 报错 → 错误日志记录到飞书表格，供运营和客服查询
- **后续**：新系统上线后，需做专门的异常信息查询功能，异常信息分类展示管理

### 结论 2：被 BLOCK 的兜底

- BLOCK 后 → 错误日志记录飞书表格
- **重试策略**：持续重试调 DW 修改接口，直至 DW 时间窗最后一天（例：DW=6.7-6.13，6.5 开始执行，一直重试到 6.13）
- **通知方式**：利用飞书表格的定时通知功能

### 结论 3：飞书表格 → TMS 数据流

**临时方案**。当前使用第三方 SaaS 系统（非开源），数据存在断层。新系统上线后：

- ship track 有独立模块
- **主流程不变**，仅取数源从"第三方系统/飞书表格"变为"新 TMS"
- 本次改动均为临时方案

### 结论 4：RDD / 数据设计 更新

基于以上结论更新 RDD §3.2-§3.3，补充第三阶段 DW 更新流程（飞书表格拉取 → API 修改 → 错误重试），标注为临时方案。
